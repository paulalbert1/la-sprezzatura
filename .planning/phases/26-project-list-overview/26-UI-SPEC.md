# Phase 26 — UI Design Contract

> Visual and interaction contract for the admin project list and project overview pages. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Astro + Tailwind CSS v4 with custom `@theme` tokens) |
| Preset | not applicable |
| Component library | Hand-built Astro components + React islands (`client:load`) for interactivity |
| Icon library | lucide-react 1.7.0 (already installed; used by AdminNav.tsx) |
| Font | Cormorant Garamond (`--font-heading`) for headings, DM Sans (`--font-body`) for body |

**Note:** This project uses Astro 6 with Tailwind CSS v4 and `@theme` directive tokens defined in `src/styles/global.css`. No shadcn, no Radix, no component.json. All UI components are hand-built. The shadcn initialization gate does not apply.

## Typography

All typography inherits from `global.css` base styles. Headings use `font-heading` (Cormorant Garamond), body text uses `font-body` (DM Sans).

| Role | Size | Weight | Line Height | Font Family |
|------|------|--------|-------------|-------------|
| Body | 14px (text-sm) | 400 (font-normal) | 1.5 | DM Sans |
| Label | 12px (text-xs) | 400 (font-normal) | 1.5 | DM Sans, uppercase tracking-widest |
| Heading | 20px (text-xl) | 600 (font-semibold) | 1.2 | DM Sans (admin page titles use body font per existing AdminLayout pattern) |
| Display | 24px (text-2xl) | 300 (font-light) | 1.2 | Cormorant Garamond (project name on overview page) |

**Rationale:** The existing AdminLayout top bar uses `text-xl font-semibold font-body` for page titles. Dashboard summary card values use `text-xl font-heading`. This phase follows both patterns: DM Sans semibold for functional headings, Cormorant Garamond light for the project display name.

## Component Inventory

### Page 1: Project List (`/admin/projects`)

**File:** `src/pages/admin/projects/index.astro` (SSR, `prerender = false`)
**React Island:** `src/components/admin/ProjectList.tsx` (`client:load`)

#### Layout Decision: Table (not card grid)

A table layout is chosen over a card grid because:
1. The project list is a management tool -- Liz needs to scan and compare across rows
2. Information density matters: pipeline stage, engagement type, client name, and status must all be visible without clicking into each project
3. The existing ProcurementTable.astro establishes this pattern for admin-facing data
4. Liz's project count is small (under 50) -- a table fits without pagination

#### Filter Pills

```
[ All ] [ Discovery ] [ Concept ] [ Design Development ] [ Procurement ] [ Installation ] [ Closeout ]
```

- Position: Above the table, left-aligned, with `gap-3` (12px) between pills
- Active pill: `bg-charcoal text-white border-charcoal` (matches existing FilterPills.astro pattern)
- Inactive pill: `bg-cream-dark text-charcoal border-stone-light hover:bg-charcoal hover:text-white hover:border-charcoal`
- Shape: `rounded-full px-5 py-2`
- Typography: `text-xs uppercase tracking-widest font-body`
- Includes "All" pill as first item (default active)
- Client-side filtering via React state (D-03)

**Rationale for charcoal pills (not terracotta):** The existing public FilterPills.astro uses charcoal for active state. The admin reuses this established pattern for consistency. Terracotta is reserved for CTAs and nav accents, not filter controls.

#### Table Structure

```
+------------------+-------------------+----------------------+----------+---------+
| Project          | Client            | Engagement           | Stage    | Status  |
+------------------+-------------------+----------------------+----------+---------+
| Flower Hill...   | Sarah & Tom...    | Full Interior Design | [badge]  | [badge] |
| Harbor View...   | James Miller      | Styling & Refreshing | [badge]  | [badge] |
+------------------+-------------------+----------------------+----------+---------+
```

| Column | Width | Content | Alignment |
|--------|-------|---------|-----------|
| Project | flex-grow (remaining space) | Title as clickable link to overview page | Left |
| Client | `w-40` | Primary client name (from `clients` reference) | Left |
| Engagement | `w-48` | Engagement type label (formatted from slug) | Left |
| Stage | `w-32` | Pipeline stage badge (colored, rounded-full) | Center |
| Status | `w-24` | Project status badge (colored, rounded-full) | Center |

- Table header: `text-xs uppercase tracking-widest text-stone font-normal pb-3 border-b-2 border-stone-light/30`
- Table rows: `border-b border-stone-light/10`, alternating `bg-cream-dark/50` on odd rows
- Project title link: `text-charcoal hover:text-terracotta transition-colors font-medium`
- Row click: entire row is NOT clickable (only the project title is a link -- avoids accidental navigation)
- Responsive: horizontal scroll on narrow viewports (`overflow-x-auto`)

## Interaction Contracts

### Filter Pill Selection

1. Page loads with "All" pill active (charcoal background)
2. User clicks a pipeline stage pill (e.g., "Procurement")
3. The clicked pill becomes active (charcoal background); all others revert to inactive (cream-dark)
4. The table filters to show only projects matching that pipeline stage
5. Filtering is instant (client-side, React state) -- no server round-trip
6. If no projects match the filter, show: "No projects in this stage" in `text-sm text-stone text-center py-8`

### Project Title Link

1. User clicks a project title in the table
2. Browser navigates to `/admin/projects/[projectId]` (overview page)
3. Standard `<a>` tag navigation -- no client-side routing

### Overview Sub-Section Cards

1. User clicks any of the four sub-section cards
2. Browser navigates to the sub-section route
3. Entire card is the click target (card is wrapped in `<a>`)
4. Hover state: border becomes slightly more visible (`border-stone-light/40`)

### Edit Project Flow

1. User clicks "Edit Project" button on overview page
2. Browser navigates to `/admin/projects/[projectId]/edit`
3. Form pre-populates with current project data
4. User modifies fields and clicks "Save Changes"
5. React island patches the project document via `writeClient.patch(projectId).set({...}).commit()`
6. On success: show "Project updated" message, then redirect back to overview page after 1.5 seconds
7. On error: show "Unable to save. Please try again." message inline above the form
8. "Cancel" link navigates back to overview page without saving (standard `<a>` to `/admin/projects/[projectId]`)

### Breadcrumb Navigation

1. Top-level pages (Dashboard, Projects list): breadcrumb prop omitted, top bar shows `pageTitle` as before
2. Detail pages: breadcrumb prop passed as array of `{label, href}` pairs
3. Last item in array has no `href` (current page, rendered as plain text)
4. All other items render as clickable links

### Dashboard Active Projects Count

1. On dashboard page load, the "Active Projects" summary card fetches the count of projects where `projectStatus == "active"`
2. The `--` placeholder is replaced with the real number
3. This uses a lightweight GROQ count query, not the full project list query

## Accessibility

| Concern | Implementation |
|---------|---------------|
| Filter pills | `role="group"` with `aria-label="Filter by pipeline stage"` on container; each pill is a `<button>` with `aria-pressed` reflecting active state |
| Table | `<table>` with `<caption class="sr-only">All projects</caption>`; proper `<thead>`, `<th scope="col">`, `<tbody>` |
| Badges | Text content is sufficient; no `aria-label` needed on badges (the text IS the label) |
| Breadcrumb | `<nav aria-label="Breadcrumb">` with `<ol>` containing `<li>` items; current page has `aria-current="page"` |
| Edit form | `<label>` elements associated with inputs via `htmlFor`/`id`; required fields have `aria-required="true"` |
| Focus visible | Inherited from `global.css`: `2px solid terracotta, offset 3px` |
| Sub-section cards | Each card is an `<a>` with descriptive text content; no additional `aria-label` needed |
