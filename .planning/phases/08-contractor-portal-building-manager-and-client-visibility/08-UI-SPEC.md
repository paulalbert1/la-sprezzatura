# Phase 8 -- UI Design Contract

Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge internal padding |
| sm | 8px | Compact element spacing, field error text margin |
| md | 16px | Default element spacing, form field gaps, card internal spacing |
| lg | 24px | Card internal padding (p-6), section heading to content gap |
| xl | 32px | Layout gaps, page horizontal padding (px-6 + responsive) |
| 2xl | 48px | Major section breaks (py-12), portal page vertical padding |
| 3xl | 64px | Page-level spacing (mt-16 on sign-out links) |

Exceptions:
- 12px (`space-y-3`, `pb-3`, `gap-3`, `mb-3`, `mt-3`): Used for compact list item spacing where 8px is too tight and 16px is too loose. Justified by existing codebase pattern -- `MilestoneSection.astro` and `ProcurementTable.astro` both use `space-y-3` and `pb-3` for list items. Multiples-of-4 compliant (12 = 3 * 4).
- 80px (`md:py-20`): Desktop page-level vertical padding on portal main content. Justified by existing codebase pattern -- every portal page uses `py-12 md:py-20` (established in Phase 5 `/portal/project/[projectId].astro`). Multiples-of-4 compliant (80 = 20 * 4).
- 44px minimum touch target height on all portal buttons (login CTA, download buttons, note submission) -- WCAG 2.2 Target Size
- Sanity Studio components use `@sanity/ui` spacing tokens, not Tailwind tokens

## Typography

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 14px-16px (`text-sm` / `text-base`) | 400 (regular) | 1.5-1.7 | DM Sans | Paragraph text at 16px (1.7 line-height); help text, empty state body, note text, form placeholders at 14px (1.5 line-height) |
| Label | 12px (`text-xs`) | 400 (regular) | 1.5 | DM Sans | Uppercase tracking-widest labels, brand mark, section metadata, badge text, timestamps, download links |
| Heading | 20px (`text-xl`) | 300 (light) | 1.2 | Cormorant Garamond | Section headings (Appointments, Scope of Work, Floor Plans, Estimate, etc.) |
| Display | 40px mobile / 48px desktop (`text-4xl md:text-5xl`) | 300 (light) | 1.2 | Cormorant Garamond | Page title (project name), dashboard greeting |

Weights used: 300 (light, headings) and 400 (normal, body). Two weights only. Emphasis is achieved through `text-charcoal` color (against `text-stone` for secondary text) and uppercase tracking-widest for labels, never a third font weight.

Source: Identical to Phase 5-7 typography. All three portal views (contractor, building manager, client) use the same luxury aesthetic.

## Component: ExpirationBadge

### Visual Specification

Pill-shaped badge showing COI certificate validity status. Follows the established `StatusBadge.astro` pattern.

```astro
<span class:list={[
  "inline-flex items-center px-3 py-1 rounded-full text-xs font-body tracking-wide",
  status === 'valid' && "bg-emerald-50 text-emerald-700",
  status === 'expiring' && "bg-amber-50 text-amber-700",
  status === 'expired' && "bg-red-50 text-red-700",
]}>
  {status === 'valid' && "Valid"}
  {status === 'expiring' && `Expires ${formattedDate}`}
  {status === 'expired' && `Expired ${formattedDate}`}
</span>
```

### Badge Logic

| Condition | Status | Color | Label |
|-----------|--------|-------|-------|
| `expirationDate > today + 30 days` | valid | emerald-50/emerald-700 | "Valid" |
| `today < expirationDate <= today + 30 days` | expiring | amber-50/amber-700 | "Expires {date}" |
| `expirationDate <= today` | expired | red-50/red-700 | "Expired {date}" |
| No expiration date set | valid | emerald-50/emerald-700 | "Valid" |

Date format: `MMM D` (e.g., "Expires Apr 15", "Expired Mar 1").

## Interaction Contract

### Contractor Work Order Detail -- State Flows

**Page load:**
```
[URL hit] --> [Middleware checks contractor session]
  --valid--> [Fetch GROQ work order detail]
    --found--> [Render page with all sections]
    --not found--> [Redirect to /workorder/dashboard]
  --invalid--> [Redirect to /workorder/login]
```

**Contractor note submission:**
```
[Collapsed trigger] --click--> [Editing form visible]
  --submit--> [Submitting, disabled] --success--> [Success message, 3s auto-collapse]
                                     \--error----> [Error banner + retry link]
  --discard--> [Collapsed]
```

**Download (floor plans, estimates, COIs, legal docs):**
```
[Click download link] --> [Browser navigates to /api/blob-serve?path={path}]
  --> [Server validates session, generates signed URL, streams file]
```

### Building Manager Auth Flow

```
[/building/login] --submit email--> [requestBuildingManagerMagicLink action]
  --success--> [Success confirmation (always, for user enumeration prevention)]
  --rate limit--> [Rate limit error]

[Email received] --click link--> [/building/verify?token={token}]
  --valid + single role--> [Create session] --> [Redirect to /building/dashboard]
  --valid + dual/triple role--> [Redirect to /portal/role-select]
  --invalid/expired--> [Redirect to /building/login?error=expired]

[/building/dashboard]
  --single project--> [Auto-redirect to /building/project/{id}]
  --multiple projects--> [Show project cards grid]
```

### Building Manager Studio Action Flow

```
[Click "Send Building Access" on project] --> [Dialog open]
  [Check: isCommercial? buildingManager.email?]
    --missing--> [Action returns null, button not shown]
    --present--> [Show dialog]
[Dialog] --click Send--> [Sending...] --success--> [Dialog auto-close]
                                      \--error----> [Console error, dialog stays open]
```

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Focus visible | `outline: 2px solid var(--color-terracotta); outline-offset: 3px` (global.css) |
| Touch targets | Minimum 44px height on all interactive elements (buttons, download links) |
| Form errors | `role="alert"` on error messages (established pattern) |
| Form labels | All inputs have associated `<label>` elements with `for`/`htmlFor` |
| Color contrast | Charcoal on cream = 12.5:1 (AAA). Stone on cream = 3.6:1 (AA large). Terracotta on white = 4.5:1 (AA). |
| COI badge contrast | Emerald-700 on emerald-50 = 4.7:1 (AA). Amber-700 on amber-50 = 4.6:1 (AA). Red-700 on red-50 = 5.6:1 (AA). |
| Download links | All download `<a>` tags include descriptive noun ("Download Plan", "Download PDF", "Download Certificate", "Download Document", "Download Estimate") -- never bare "Download" or icon-only |
| Telephone links | `href="tel:{number}"` on phone numbers for mobile tap-to-call |
| Email links | `href="mailto:{email}"` on email addresses |
| Screen reader | COI badges include status text (not color-only). Past appointments communicated via text "past" in sr-only span or opacity provides supplementary cue. |
| Keyboard nav | Tab order follows visual top-to-bottom order. All interactive elements reachable via keyboard. |
| Reduced motion | No animations on portal pages (static layouts, no GSAP) |
| noindex/nofollow | All portal pages include `<meta name="robots" content="noindex, nofollow">` via PortalLayout.astro |
| Table semantics | COI and legal doc lists use semantic markup (not `<table>` -- card layout with proper heading hierarchy) |
