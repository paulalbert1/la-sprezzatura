Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing, field error text margin |
| md | 16px | Default element spacing, form field gaps, Studio dialog padding |
| lg | 24px | Card internal padding (p-6), section padding |
| xl | 32px | Layout gaps, page horizontal padding |
| 2xl | 48px | Major section breaks, portal page vertical padding |
| 3xl | 64px | Page-level spacing (top/bottom of portal main content) |

Exceptions:
- 44px minimum touch target height on login button and dashboard project cards (WCAG 2.2 Target Size)
- Studio components use `@sanity/ui` spacing tokens (space array: 0-5), not Tailwind tokens -- these are managed by the Sanity design system and do not need explicit declaration here

## Typography

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 16px (`text-base`) | 400 (regular) | 1.7 | DM Sans | Paragraph text, help text, placeholder dashboard body |
| Label | 12px (`text-xs`) | 400 (regular) | 1.5 | DM Sans | Uppercase tracking-widest labels, brand mark, engagement type label, metadata |
| Heading | 24px (`text-2xl`) | 300 (light) | 1.2 | Cormorant Garamond | Section headings, confirmation headings |
| Display | 40px mobile / 48px desktop (`text-4xl md:text-5xl`) | 300 (light) | 1.2 | Cormorant Garamond | Login page heading, dashboard greeting |

Weights used: 300 (light, headings) and 400 (normal, body). Two weights only. Emphasis is achieved through color contrast (`text-charcoal` vs `text-stone`) and letter-spacing, never a third font weight.

Source: Identical to Phase 5 typography. Contractor portal pages use the same luxury aesthetic and font sizing as client portal pages (CONTEXT.md locked decision).

Identical to `/portal/login` layout. Centered single-column form, max-width 400px, vertically centered in viewport. Wrapped in `PortalLayout.astro`.

### Visual Treatment

Every visual property matches the client login page exactly:
- Background: cream

The only differences from the client login page are the copy strings (see Copywriting Contract).

## Sanity Studio: BlobFileInput Component

### Visual Specification

Uses `@sanity/ui` components exclusively. Follows the `PortalUrlDisplay.tsx` pattern.

**Idle state (no file uploaded):**
```
Stack (space=3)
  Flex (gap=2)
    <input type="file"> styled with Button (text="Choose File", mode="ghost")
  Text (size=1, muted): "Upload a PDF, JPEG, PNG, or WebP file"
```

**Uploading state:**
```
Stack (space=3)
  Flex (gap=2, align="center")
    Spinner (muted)
    Text (size=1): "Uploading..."
```

**Uploaded state (file stored):**
```
Stack (space=3)
  Flex (gap=2, align="center")
    Text (size=1): "{filename}" (truncated to 40 chars with ellipsis)
    Button (text="Remove", tone="critical", mode="ghost", fontSize=1)
```

**Error state:**
```
Stack (space=3)
  Card (tone="critical", padding=3, radius=2)
    Text (size=1): "Upload failed: {error message}"
  Button (text="Try Again", mode="ghost", fontSize=1)
```

### Accepted File Types
- `application/pdf`
- `image/jpeg`
- `image/png`
- `image/webp`

### Behavior
- File picker opens native OS dialog
- Upload happens immediately on file selection (no separate "Upload" button)
- On success, blob pathname is stored in the Sanity string field via `onChange(set(blob.pathname))`
- Remove button calls `onChange(unset())` -- does NOT delete the blob from Vercel Blob (orphan cleanup is a future concern)
- Maximum file size: enforced by Vercel Blob (default 4.5MB on Hobby, 500MB on Pro). No client-side validation needed.

Uses inline styles (established pattern from `notifyClient.tsx`). Appears on contractor documents only.

**Action button (in Studio toolbar):**
- Label: "Send Work Order Access"

## Interaction Contract

### Contractor Login Flow States

```
[Idle] --submit--> [Submitting] --success--> [Success Confirmation]
                               \--error----> [Error Banner + Idle Form]
```

Identical interaction model to client login (Phase 5):
- React component with `client:load`, calls contractor-specific Astro Action
- On submit: disable button, show "Sending..." text
- On success: replace form with confirmation message (no redirect)
- On error: show error banner above form, re-enable button
- "Email not found" treated as success (user enumeration prevention)
- Rate limit responses show rate limit error message

### Contractor Verify Flow

```
[Token consumed] --valid--> [Check dual-role]
                                   |
                            [Single role] --> [Create session] --> [Redirect to /workorder/dashboard]
                            [Dual role]   --> [Redirect to role selection page with token]

[Token consumed] --invalid--> [Redirect to /workorder/login?error=expired]
```

### Role Selection Flow

```
[Role selection page] --click Client--> [Create client session] --> [Redirect to /portal/dashboard]
[Role selection page] --click Contractor--> [Create contractor session] --> [Redirect to /workorder/dashboard]
```

Standard form submissions (POST). No client-side JavaScript required.

### Blob Upload Flow (Studio)

```
[No file] --select file--> [Uploading] --success--> [File uploaded, pathname stored]
                                       \--error----> [Error message + retry]
[File uploaded] --click Remove--> [No file]
```

- Upload begins immediately on file selection
- No separate confirmation step
- Pathname stored in Sanity string field

### Studio Action Flow (SendWorkOrderAccess)

```
[Click action button] --> [Dialog open]
[Dialog] --click Send--> [Sending...] --success--> [Dialog auto-close]
                                      \--error----> [Console error, dialog stays open]
```

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Focus visible | `outline: 2px solid var(--color-terracotta); outline-offset: 3px` (established in global.css) |
| Touch targets | Minimum 44px height on login button and dashboard cards |
| Form errors | `role="alert"` on error messages (established LoginForm pattern) |
| Form labels | All inputs have associated `<label>` elements with `for`/`htmlFor` |
| Color contrast | Charcoal on cream = 12.5:1 (AAA). Stone on cream = 3.6:1 (AA for large text). Terracotta on white = 4.5:1 (AA). |
| Screen reader | Role selection buttons have descriptive text (label + description). |
| Keyboard nav | Tab order: email input then submit button. Enter submits form. Role selection navigable via Tab + Enter. |
| Reduced motion | No animations on portal pages (portal uses static layouts, no GSAP) |
| noindex/nofollow | All contractor pages include `<meta name="robots" content="noindex, nofollow">` via PortalLayout.astro |

## Sanity Studio Field Gating -- Visual Reference

This section documents the expected Studio UX for engagement type and commercial toggle gating. These are not custom components -- they use Sanity's built-in `hidden` callback on fields.

### Engagement Type Tab Visibility

| Tab | Full Interior Design | Styling & Refreshing | Carpet Curating |
|-----|---------------------|----------------------|-----------------|
| Content | Visible | Visible | Visible |
| Portal | Visible | Visible | Visible |
| Milestones | Visible | Visible | Visible |
| Procurement | Visible | Hidden (empty tab) | Hidden (empty tab) |
| Artifacts | Visible | Visible | Visible |
| Contractors | Visible | Hidden (empty tab) | Hidden (empty tab) |

Note: Due to a Sanity limitation (group-level `hidden` does not receive `document` parameter -- GitHub #3305), gating is applied at the field level. When all fields in a group are hidden, the tab still appears but shows empty content. This is acceptable Studio UX. (See RESEARCH.md Pitfall 5.)

### Commercial Toggle

| Field/Section | Residential (default) | Commercial |
|---------------|----------------------|------------|
| Building Manager (name, email, phone) | Hidden | Visible |
| COIs array | Hidden | Visible |
| Legal Documents array | Hidden | Visible |

These fields live in the Portal tab (RESEARCH.md recommendation). The `isCommercial` boolean checkbox ("Commercial Project") is in the Content tab alongside engagement type.
