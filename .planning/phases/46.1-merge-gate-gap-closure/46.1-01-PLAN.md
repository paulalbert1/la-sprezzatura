```ts
// Strip a single leading "Hi {firstName}{,|!}?\n+" occurrence to prevent the
// double-greeting collision when admin types personalNote that starts with the
// same greeting the structural Greeting component already emits.
//
// Narrow by design (D-1): case-sensitive "Hi", single-word firstName, optional
// `,` or `!`, ≥1 trailing newline. Does NOT match "Hello", "Dear", "Hey", or
// lowercase "hi". Does NOT iterate — strips ONE leading match only. Does NOT
// match mid-paragraph occurrences (anchored at start).
const LEADING_GREETING_RE = /^Hi\s+\w+[,!]?\s*\n+/;

export function stripLeadingGreeting(personalNote: string): string {
  return personalNote.replace(LEADING_GREETING_RE, "");
}
```

Then inside the existing `Body` function, change:
```ts
if (!personalNote.trim()) return null;
```

```ts
const stripped = stripLeadingGreeting(personalNote);
if (!stripped.trim()) return null;
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin compose UI → email render pipeline | Authenticated admin authors personalNote string; flows through Sanity (or directly via send) into the email render. The string is rendered as HTML (after markdown serialization). |
| email render → recipient inbox | Rendered HTML reaches an external mail client. |

Both boundaries already existed before this plan. The plan adds a single string-transform (`stripLeadingGreeting`) on the input side; it does not introduce a new trust boundary.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46.1-01-01 | D (DoS — ReDoS) | `LEADING_GREETING_RE` regex applied to admin-authored personalNote in `src/emails/sendUpdate/Body.tsx` | mitigate | The regex is anchored at start (`^`), the variable-length subpattern is bounded (`\w+` matches a single word with no nested quantifiers), and there is no alternation or backtracking-prone construct. The regex runs once per render (no iteration). Worst-case input ("Hi " + 600 word characters + "," + newlines) completes in microseconds — well-known regex-engine behavior for anchored bounded patterns. Plus: personalNote is upstream-capped at 600 characters by `parsePersonalNote` (per Phase 46-04 D-7), bounding worst-case runtime structurally. No mitigation beyond the existing anchoring + bounded quantifier needed. |
| T-46.1-01-02 | T (Tampering — content elision) | `stripLeadingGreeting` removes content from admin-authored input before render | accept | The strip is intentionally narrow (single anchored leading occurrence matching exactly the structural Greeting's emit shape). It removes only what would otherwise visually duplicate the structural greeting. If admin genuinely intended to start the body with "Hi {Name}," followed by a body paragraph, the result is indistinguishable from what the structural Greeting renders — no information loss. The 12-test negative-case battery (Hello/Dear/hi/Hey/Hi everyone/mid-paragraph) confirms the strip cannot misfire on prose that happens to start with similar-looking copy. |
| T-46.1-01-03 | I (Information disclosure via XSS) | New helper paragraph in `SendUpdateModal.tsx` renders the literal string `{firstName}` | accept | The helper paragraph is a static React JSX element with no dynamic input — the `{"{firstName}"}` expression renders a string literal, not an interpolated value. There is no path from user input to this paragraph. Standard React string-children escaping handles `&quot;` and the em-dash safely. |
</threat_model>
