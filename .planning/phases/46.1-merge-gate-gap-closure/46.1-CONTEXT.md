## Decisions (locked)

### D-1 — Defensive strip for greeting collision is the safety net; admin UX helper is the prevention. Ship both.

**Why:** Defensive strip handles the immediate test-send today and any past Sanity drafts where personalNote starts with "Hi {Name},". Admin UX helper prevents the habit from re-forming. Either alone is insufficient — strip-only leaves users mystified about why their typed greeting "disappears", helper-only fails when users skip past the helper text. Both together are tight.

**Strip pattern (locked):** `^Hi\s+\w+[,!]?\s*\n+`

- Anchored at the very start of personalNote
- Matches "Hi" literal (exact case — never strip "hi", "hello", "hey" — too aggressive)
- Single word after "Hi" (the firstName)
- Optional comma or exclamation point
- One or more trailing newlines (the line that separates greeting from body)

Strips ONE leading match. Does not iterate. Does not strip mid-paragraph occurrences. Does not strip greetings that don't follow the literal "Hi {Name}," shape (e.g., "Dear Victoria,", "Hello Victoria,", "Hi everyone,"). The pattern is deliberately narrow — it only matches the exact shape the structural Greeting component emits, so legitimate prose that happens to start with "Hi {someone}," but is NOT the doubled-greeting case never matches.

**Where the strip lives:** `src/emails/sendUpdate/Body.tsx` (or new helper colocated with Body). NOT in `SendUpdate.tsx` composition root — keep composition root declarative; mutation logic belongs in the section component that owns the personalNote rendering.

**Helper-text copy (locked):** `Don't include "Hi {firstName}," — it's added automatically.`

The helper renders below the personalNote textarea in the admin compose UI. firstName interpolates the live recipient first name when known (compose UI has access to selected client); falls back to literal `Hi {firstName}` when no client is selected yet (e.g., during template-edit before recipient pick).

### D-2 — `formatDate` / `formatLongDate` guard returns empty string `""`, not a placeholder

**Why:** Empty string renders as a quiet empty cell in the procurement table. A placeholder ("—", "TBD", "n/a") commits to a specific UX that's better resolved at the section component level if needed (Procurement.tsx could later choose to render "TBD" for missing ETAs, but that's a section-component decision, not a formatter decision). Formatter's job is "format a date or get out of the way." `""` is the cleanest get-out-of-the-way return.

**Guard order (locked):**
```ts
export function formatDate(d: string | Date): string {
  if (!d) return "";                                // null, undefined, ""
  const date = typeof d === "string" ? new Date(d) : d;
  if (isNaN(date.getTime())) return "";             // unparseable string, etc.
  return date.toLocaleDateString("en-US", { month: "short", day: "numeric" });
}
```

Apply identical guard to `formatLongDate` at the same file. Same return-empty contract.

**Tests (locked):**
- `formatDate("")` → `""`
- `formatDate(undefined as unknown as string)` → `""` (guards against typescript-bypass)
- `formatDate("not-a-date")` → `""`
- `formatDate("2026-05-14")` → `"May 14"` (regression-protect existing behavior)
- `formatDate(new Date("2026-05-14"))` → `"May 14"` (regression-protect Date overload)
- Mirror four tests for `formatLongDate` (same inputs, expect either `""` or "May 14, 2026" / "April 28, 2026" pattern).

### D-6 — Force light-mode rendering in Outlook desktop via `[data-ogsc]` / `[data-ogsb]` selectors + MSO conditional

**Why:** `EmailShell.tsx:72-73` already declares `<meta name="color-scheme" content="only light">` and `<meta name="supported-color-schemes" content="light">`. New Outlook (web/M365) and Apple Mail respect those tags and skip auto-darken. **Outlook desktop ignores them** and runs its own auto-darken pass via the `[data-ogsc]` (text-color) and `[data-ogsb]` (background-color) selector hooks, which only spare colors the email author has explicitly locked through those same selectors. The result Liz saw — dark gray body + still-light text — is the half-inverted state that emerges when the author has locked nothing. Lock the existing palette against inversion; do NOT design a dark variant.

**Edit scope (locked):**

In `src/emails/shell/EmailShell.tsx` `<Head>` (immediately after the existing two `<meta>` tags), add a `<style>` block containing:

1. **`[data-ogsc]` text-color overrides** for every text color the templates use:
   - `[data-ogsc] .text-stone, [data-ogsc] .text-stone-light` → re-pin to `#8A8478` / `#B8AFA4`
   - `[data-ogsc]` rules for inline-styled text colors `#2C2926` (dark text), `#4A4540` (mid text), `#9B3A2A` (procurement pending pill text), and the seven `STATUS_PILL_STYLES.*.text` values
   - Use `color: ... !important` on each rule (Outlook auto-darken specificity requires `!important`)

2. **`[data-ogsb]` background-color overrides** for every background:
   - `[data-ogsb] body, [data-ogsb] .bg-cream` → `background-color: #FAF8F5 !important`
   - `[data-ogsb]` rules for the seven `STATUS_PILL_STYLES.*.bg` values
   - Border colors via `[data-ogsc]` since Outlook auto-darken treats border as text-adjacent: `border-color: #E8DDD0 !important` for the cream-dark divider; pill borders via the seven `STATUS_PILL_STYLES.*.border` values
   - CTA backgrounds: terracotta `#C4836A` (SU) and gold `#9A7B4B` (WO)

3. **MSO conditional block** for older Outlook desktop builds where `[data-ogs*]` doesn't reach:
   ```html
   <!--[if mso]>
   <style type="text/css">
     body { mso-color-scheme: light !important; }
   </style>
   <![endif]-->
   ```

**Source of truth for color values:** import from existing tokens — do NOT hardcode a parallel palette. The CSS rules can list the literal hex values (CSS doesn't support imports here), but the values must be byte-identical to:
- Cream/stone/dark-text: existing inline styles in `EmailShell.tsx`, `Greeting.tsx`, `Body.tsx`, `Milestones.tsx`, `Procurement.tsx`, `ReviewItems.tsx`
- Pill chrome: `src/lib/procurement/statusPills.ts` (7 statuses × 3 properties = 21 values)
- CTA: `src/emails/shell/EmailButton.tsx` (terracotta + gold from `_theme.ts`)

If the planner finds palette drift between sources, that's a separate finding — STOP and surface it before locking the D-6 CSS values.

**Where the style lives:** inside `<Head>` of `EmailShell.tsx`, NOT a separate file. Locating the override in the shared shell ensures both SendUpdate and WorkOrder benefit; locating it inline (vs. external stylesheet) is required because email clients strip `<link rel="stylesheet">`.

### D-17 — Delete the `[data-ogsc] * { color: inherit !important }` catch-all

**Why:** With D-16's text-rule deletion, no per-class rule re-pins inline text colors after the catch-all overrides them. Inline `style="color:..."` (no `!important`) loses to `[data-ogsc] * { color: inherit !important }` per cascade. Result: every inline text color (procurement pill text on the surviving D-15 elements, body prose text, headings) goes inherit. The catch-all's intended belt-and-suspenders role evaporated when the rest of the lock turned out to be dead. Deleting it lets inline non-`!important` colors win by default — matching what we want for "force light mode."

**Per-class pill rules with `!important` (added by D-15's binding) still re-pin pill chrome** in Outlook desktop dark mode. The `@media (prefers-color-scheme: dark)` block from 46.1-08 covers webmail clients honoring the scheme. Body-bg paint via `<table bgcolor>` + inline body style + `[data-ogsb] body` rule is unchanged.

**Edit scope (locked):** Delete `EmailShell.tsx:119` (the `[data-ogsc] * { color: inherit !important; }` rule). Delete the surrounding "catch-all auto-darken cancellation" comment block.
