## Decisions (locked)

### D-1 — Defensive strip for greeting collision is the safety net; admin UX helper is the prevention. Ship both.

**Why:** Defensive strip handles the immediate test-send today and any past Sanity drafts where personalNote starts with "Hi {Name},". Admin UX helper prevents the habit from re-forming. Either alone is insufficient — strip-only leaves users mystified about why their typed greeting "disappears", helper-only fails when users skip past the helper text. Both together are tight.

**Strip pattern (locked):** `^Hi\s+\w+[,!]?\s*\n+`

- Anchored at the very start of personalNote
- Matches "Hi" literal (exact case — never strip "hi", "hello", "hey" — too aggressive)
- Single word after "Hi" (the firstName)
- Optional comma or exclamation point
- One or more trailing newlines (the line that separates greeting from body)

Strips ONE leading match. Does not iterate. Does not strip mid-paragraph occurrences. Does not strip greetings that don't follow the literal "Hi {Name}," shape (e.g., "Dear Victoria,", "Hello Victoria,", "Hi everyone,"). The pattern is deliberately narrow — it only matches the exact shape the structural Greeting component emits, so legitimate prose that happens to start with "Hi {someone}," but is NOT the doubled-greeting case never matches.

**Where the strip lives:** `src/emails/sendUpdate/Body.tsx` (or new helper colocated with Body). NOT in `SendUpdate.tsx` composition root — keep composition root declarative; mutation logic belongs in the section component that owns the personalNote rendering.

**Helper-text copy (locked):** `Don't include "Hi {firstName}," — it's added automatically.`

The helper renders below the personalNote textarea in the admin compose UI. firstName interpolates the live recipient first name when known (compose UI has access to selected client); falls back to literal `Hi {firstName}` when no client is selected yet (e.g., during template-edit before recipient pick).

### D-2 — `formatDate` / `formatLongDate` guard returns empty string `""`, not a placeholder

**Why:** Empty string renders as a quiet empty cell in the procurement table. A placeholder ("—", "TBD", "n/a") commits to a specific UX that's better resolved at the section component level if needed (Procurement.tsx could later choose to render "TBD" for missing ETAs, but that's a section-component decision, not a formatter decision). Formatter's job is "format a date or get out of the way." `""` is the cleanest get-out-of-the-way return.

**Guard order (locked):**
```ts
export function formatDate(d: string | Date): string {
  if (!d) return "";                                // null, undefined, ""
  const date = typeof d === "string" ? new Date(d) : d;
  if (isNaN(date.getTime())) return "";             // unparseable string, etc.
  return date.toLocaleDateString("en-US", { month: "short", day: "numeric" });
}
```

Apply identical guard to `formatLongDate` at the same file. Same return-empty contract.

**Tests (locked):**
- `formatDate("")` → `""`
- `formatDate(undefined as unknown as string)` → `""` (guards against typescript-bypass)
- `formatDate("not-a-date")` → `""`
- `formatDate("2026-05-14")` → `"May 14"` (regression-protect existing behavior)
- `formatDate(new Date("2026-05-14"))` → `"May 14"` (regression-protect Date overload)
- Mirror four tests for `formatLongDate` (same inputs, expect either `""` or "May 14, 2026" / "April 28, 2026" pattern).
