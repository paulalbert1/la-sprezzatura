```ts
export function formatDate(d: string | Date): string {
  if (!d) return "";                                                   // null, undefined, ""
  const date = typeof d === "string" ? new Date(d) : d;
  if (isNaN(date.getTime())) return "";                                // unparseable string, etc.
  return date.toLocaleDateString("en-US", { month: "short", day: "numeric" });
}

export function formatLongDate(d: string | Date): string {
  if (!d) return "";
  const date = typeof d === "string" ? new Date(d) : d;
  if (isNaN(date.getTime())) return "";
  return date.toLocaleDateString("en-US", { month: "long", day: "numeric", year: "numeric" });
}
```

Critical:
- The two `if (!d) return "";` early returns must precede the `new Date(d)` conversion. Order matters: `new Date("")` returns an invalid Date but `!""` is `true`, so the first guard short-circuits — keeping the explicit empty check makes intent obvious for future readers and avoids relying on the second guard to catch the empty-string case.
- Use `isNaN(date.getTime())` (the canonical invalid-Date probe). Do NOT use `Number.isNaN` here — `Number.isNaN(date.getTime())` would behave identically for the value but `isNaN` is the convention in the existing codebase for this idiom.
- The `.toLocaleDateString()` calls and their format options stay byte-identical to the current code — `{ month: "short", day: "numeric" }` for `formatDate` and `{ month: "long", day: "numeric", year: "numeric" }` for `formatLongDate`. Do NOT touch the format options.
