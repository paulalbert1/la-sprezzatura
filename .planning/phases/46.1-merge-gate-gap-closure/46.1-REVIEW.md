## Summary

Phase 46.1 closes five UAT gaps from Phase 46's Outlook merge-gate test (gap-1 double-greeting strip, gap-2 empty-date guard, gap-4 dark-mode lock, gap-5 left-align table cells; gap-3 was docs-only and out of this review's file scope). The implementation lines up tightly with the locked decisions in `46.1-CONTEXT.md` and the test coverage on each fix is genuinely load-bearing, not vacuous.

No BLOCKERs found. The bug fixes for gap-1, gap-2, and gap-5 are correct and the assertions hold up under trace-the-input. The dark-mode lock (gap-4) is well-tested at the rendered-HTML level and matches the planned pattern.

That said, several WARNINGs are worth surfacing before closing the parent UAT:

- The `stripLeadingGreeting` regex is narrower than its intended job in two practical ways (hyphenated/apostrophe firstNames will still double-greet; generic salutations like "Hi all," will be silently elided when the structural greeting is "Hi Sarah,"). The regex shape is locked by D-1, so this is a constraint to acknowledge — but it should be visible to the parent UAT re-test.
- The admin compose UI does not enforce the 600-char `personalNote` cap that `parsePersonalNote` enforces. Admin can type 700 chars, click Preview or Send, and hit a `PersonalNoteParseError` that propagates as an opaque 500 from the API/preview route.
- The `[data-ogsc] * { color: inherit !important; }` catch-all rule in `EmailShell.tsx` is broader than the surrounding scoped rules and risks cancelling intentional inline color in any future section component.
- `EmailShell.tsx` injects an MSO conditional comment via `<div style={{display:none}}>` inside `<head>`. `<div>` is not valid in `<head>`; relying on react-email + Outlook's lax parsers is fragile.
- The `borderRadius: "2px"` in `Procurement.tsx` line 90 is shorthand, but the inline comment claims "longhand only -- Phase 46 D-3". The code or the comment is wrong; either way the intent is unclear for future readers.

Info items below cover smaller quality issues (test description mismatches, hardcoded tenant strings, stale-state risks in modal, fragile regex composition).

## Warnings

### WR-01: `stripLeadingGreeting` does not handle hyphenated/apostrophe firstNames — gap-1 still leaks for those clients

**File:** `src/emails/sendUpdate/Body.tsx:23`
**Issue:** The locked regex `^Hi\s+\w+[,!]?\s*\n+` uses `\w+`, which matches `[A-Za-z0-9_]` only. Real client firstNames like `Mary-Anne`, `O'Brien`, `Anne Marie` will not be caught. If admin types `"Hi Mary-Anne,\n\nbody"` and the structural Greeting (`Greeting.tsx:59`) emits `"Hi Mary-Anne,"`, the email will still double-greet — exactly the gap-1 failure mode the parent UAT diagnosed. The CONTEXT.md D-1 wording ("single-word firstName") presupposes ASCII-alphanumeric names, but the production Sanity dataset is not constrained that way.

This is a locked-by-D-1 narrowness rather than a regression introduced here, but the parent UAT (`46-UAT.md`) re-test should explicitly include a hyphenated-name case before resolving gap-1, otherwise the fix passes UAT for "Victoria" and silently fails the same defect for any compound-name client.

**Fix:** Either widen the character class to cover the realistic firstName set (`[\w'-]+`, kept anchored at start, still narrow):
```ts
const LEADING_GREETING_RE = /^Hi\s+[\w'\-]+[,!]?\s*\n+/;
```
…or document the limitation explicitly in `Body.tsx` and add a test asserting the hyphenated-name case is intentionally NOT stripped (so the next reader knows the strip's blind spot is by design, not by oversight). Surface the choice in 46.1's UAT replay before closing the parent.

### WR-02: `stripLeadingGreeting` matches ANY single word, eliding generic salutations the admin may have intended

**File:** `src/emails/sendUpdate/Body.tsx:23`, `src/emails/sendUpdate/Body.test.ts:46-69`
**Issue:** The regex doesn't validate that the matched word equals `clientFirstName`. So `"Hi everyone,\n\nbody"`, `"Hi all,\n\nbody"`, `"Hi team,\n\nbody"` all get the leading line silently elided when the structural Greeting is `"Hi Sarah,"`. The recipient sees `"Hi Sarah,"` followed by the body — the admin's intentional "Hi all," opener is gone with no log, no error, no warning.

The negative-case test on line 66-69 catches the multi-word `"Hi everyone, this is the body"` but only because there's no trailing newline. Once admin adds the natural blank-line break after their salutation, the strip fires.

This is technically defensible (D-1 says "narrow by design — only the doubled-greeting case"), but it's a content-elision risk: the admin can lose intentional opening text without feedback. The test suite does not cover this scenario at all.

**Fix:** Add a defensive test asserting the behavior is intentional (so future readers see it):
```ts
it("DOES strip 'Hi all,' even when greetingName != 'all' (known false-positive, narrow-by-design)", () => {
  expect(stripLeadingGreeting("Hi all,\n\nBody.")).toBe("Body.");
});
```
Or — and this is the safer functional fix — pass `clientFirstName` to `stripLeadingGreeting` and only strip when the captured word equals it (case-insensitive). That tightens the strip to the exact double-greeting case CONTEXT.md actually describes.

### WR-03: SendUpdateModal does not enforce the 600-char `personalNote` cap that `parsePersonalNote` enforces — admin hits cryptic errors

**File:** `src/components/admin/SendUpdateModal.tsx:451-463`
**Issue:** `personalNoteMarkdown.ts:23` declares `MAX_LEN = 600` and throws `PersonalNoteParseError("OVER_LIMIT")` when exceeded. `Body.tsx` lets the error propagate ("Body lets the error propagate" — file header). The modal's `<textarea>` (line 451-463) has no `maxLength` attribute, no character count, no warning text. Admin types 700 chars → clicks Preview → preview route renders `<SendUpdate>` → `parsePersonalNote` throws → 500 response in the new tab. Same path on Send: POST returns whatever the API does with the thrown error (best case a friendly error string in the response body, worst case a 500). Either way the failure mode is invisible until the admin tries to send.

This is not the gap-1 fix landing wrong — it's pre-existing — but the gap-1 fix added the `Don't include "Hi {firstName}," — it's added automatically.` helper line right next to the textarea, which is the natural home for a length indicator. Worth catching before merge-gate replays.

**Fix:** Add `maxLength={600}` to the `<textarea>` and a small character counter below it:
```tsx
<textarea
  id="send-update-note"
  rows={4}
  maxLength={600}
  className="luxury-input w-full"
  ...
/>
<p style={{...}}>
  {personalNote.length} / 600 characters
</p>
```
The 600 should ideally come from `personalNoteMarkdown.ts` as an exported constant rather than being duplicated, so changing the cap in one place changes both checks.

### WR-04: `[data-ogsc] * { color: inherit !important; }` catch-all is broader than its surrounding scoped rules and risks future regressions

**File:** `src/emails/shell/EmailShell.tsx:119`
**Issue:** The universal-selector rule applies to every descendant element under any ancestor with `data-ogsc`. The comment on line 115-118 frames it as "catch-all auto-darken cancellation". The risk: any future section component that relies on a literal inline `style="color:..."` to render a non-inherited color (e.g., a status callout, a warning row) will silently lose its color in Outlook desktop because `* { color: inherit !important }` outranks the inline color via `!important`.

Per CSS specificity rules, an ID selector or higher-specificity class would beat this, but inline style with `!important` would not (because `[data-ogsc] *` plus `!important` matches inline `!important` and resolves cascade by source order; the head-style block is parsed before the body, so head wins on equal-specificity ties when both are `!important` — actually, inline styles have higher specificity than universal even with `!important`, so inline `!important` wins. This is enough of a corner case that it's not obvious to a future developer reading the CSS.).

The narrower fix is to scope the rule to the children that actually need it, not all descendants. Today, the per-class rules above already cover the known palette; the catch-all is a belt-and-suspenders that may overshoot.

**Fix:** Remove the universal-selector rule and trust the scoped rules above; if a future section component adds inline color, lock it via a new `[data-ogsc] .new-class` rule and re-run the merge-gate test in Outlook desktop. If the universal rule is genuinely load-bearing for inline-styled text the planner observed in UAT, leave it in but add a comment naming the specific element it was inserted to protect, so future readers know what regression to watch for.

### WR-05: MSO conditional comment is wrapped in a `<div>` inside `<head>` — invalid HTML, fragile across react-email versions

**File:** `src/emails/shell/EmailShell.tsx:138-143`
**Issue:** The conditional Outlook comment is injected via `<div style={{display:none}} dangerouslySetInnerHTML={...}/>` placed inside `<Head>`. Per HTML5 (and HTML4 before it), `<div>` is not a valid child of `<head>`; only metadata content (title, base, link, meta, noscript, script, style, template) is allowed. The browser's HTML parser will move the `<div>` to `<body>` during parsing, which means the conditional comment may end up in the wrong place once a webmail client re-parses the rendered HTML.

The current rendered output happens to keep the `<div>` in head because react-email's `render()` uses a non-spec-conformant string serializer (it doesn't fix-up parse errors), so the bug is latent — it works today, it could stop working on the next react-email version bump.

**Fix:** Inline the MSO conditional inside the existing `<style>` element on line 82-133. The MSO conditional comment can live above or below the existing CSS rules in the same `<style dangerouslySetInnerHTML>` payload — Outlook desktop processes `<!--[if mso]>` boundaries inside `<style>` tags in older builds. Verify the rendered HTML still contains the conditional in the merge-gate UAT replay. Alternatively, inject the conditional as the second `<style>` element in the head with its CSS payload guarded:
```tsx
<style dangerouslySetInnerHTML={{
  __html: `<!--[if mso]>body { mso-color-scheme: light !important; }<![endif]-->`
}} />
```
Test that EmailShell.test.ts's assertions on `<!--[if mso]>` and `<![endif]-->` still pass (both substring-only).

### IN-04: `SendUpdateModal.tsx` `useState(() => showProcurementRow)` captures initial value only — stale if `project` prop changes

**File:** `src/components/admin/SendUpdateModal.tsx:206-208`
**Issue:** The lazy initializer `useState(() => showProcurementRow)` captures the boolean at modal-mount time. If the parent passes a new `project` prop while the modal is open (e.g., projectId switches in the URL without unmount), `showProcurementRow` updates via `useMemo` but `includeProcurement` keeps its initial value. Practically this is uncommon — most modals unmount on close — but the surrounding state pattern (no `useEffect` to sync) makes the assumption implicit.

### IN-05: `Body.tsx` regex `\s*\n+` is fragile due to overlap between `\s` and `\n`

**File:** `src/emails/sendUpdate/Body.tsx:23`
**Issue:** `\s*\n+` is technically valid because regex backtracks, but `\s` includes `\n`, so the engine has to try-and-fail combinations. For `"Hi Sarah,\n"` the regex passes via backtracking; for `"Hi Sarah,\n\n"` likewise. The intent appears to be "optional trailing whitespace on the greeting line, then >=1 newline" — `[ \t]*\n+` expresses that without ambiguity.

Not a bug today, but the next regex maintainer will read `\s*\n+` and pause to verify it works.

**Fix:**
```ts
const LEADING_GREETING_RE = /^Hi\s+\w+[,!]?[ \t]*\n+/;
```

### IN-06: `SendUpdate.tsx` `sentDate ?? formatLongDate(new Date())` falls through for empty string

**File:** `src/emails/sendUpdate/SendUpdate.tsx:156`
**Issue:** `??` only fallbacks for `null`/`undefined`. If a caller passes `sentDate: ""` (which the type allows — `sentDate?: string` so empty string is permitted), `Greeting` renders `{title} · ` with a trailing dot and an empty space. The post-46.1 `formatDate`/`formatLongDate` guards return `""` for empty/invalid input, which makes a downstream caller chaining `formatDate(somefield)` into `sentDate` more likely to produce empty strings.

**Fix:** Change to a truthy check:
```ts
const sentDateFormatted = sentDate || formatLongDate(new Date());
```
Or guard the type more tightly: `sentDate?: string | undefined` and trim-check at the boundary.

### IN-07: `Procurement.tsx` and `Milestones.tsx` use array-index keys (`m-${i}` / `p-${i}`)

**File:** `src/emails/sendUpdate/Milestones.tsx:101`, `src/emails/sendUpdate/Procurement.tsx:111`
**Issue:** Standard React anti-pattern: index-based keys break reconciliation if the list is sortable/filterable. For email render, the list is mutated only at compose time (server-side render, no re-render after mount), so this is benign in practice. Worth noting because `MilestoneRow`/`ProcurementRow` don't carry stable IDs — if a future feature adds row-level animation or interaction, the keys will need to change.

**Fix:** Either accept index-keys with a comment explaining why they're safe in email-render (server-only, no re-render), or thread a stable `_key`/`id` through the row types from the Sanity source. The latter aligns with how `pendingArtifacts` already carries `_key`.
