### WR-01: `stripLeadingGreeting` does not handle hyphenated/apostrophe firstNames — gap-1 still leaks for those clients

**File:** `src/emails/sendUpdate/Body.tsx:23`
**Issue:** The locked regex `^Hi\s+\w+[,!]?\s*\n+` uses `\w+`, which matches `[A-Za-z0-9_]` only. Real client firstNames like `Mary-Anne`, `O'Brien`, `Anne Marie` will not be caught. If admin types `"Hi Mary-Anne,\n\nbody"` and the structural Greeting (`Greeting.tsx:59`) emits `"Hi Mary-Anne,"`, the email will still double-greet — exactly the gap-1 failure mode the parent UAT diagnosed. The CONTEXT.md D-1 wording ("single-word firstName") presupposes ASCII-alphanumeric names, but the production Sanity dataset is not constrained that way.

This is a locked-by-D-1 narrowness rather than a regression introduced here, but the parent UAT (`46-UAT.md`) re-test should explicitly include a hyphenated-name case before resolving gap-1, otherwise the fix passes UAT for "Victoria" and silently fails the same defect for any compound-name client.

**Fix:** Either widen the character class to cover the realistic firstName set (`[\w'-]+`, kept anchored at start, still narrow):
```ts
const LEADING_GREETING_RE = /^Hi\s+[\w'\-]+[,!]?\s*\n+/;
```
…or document the limitation explicitly in `Body.tsx` and add a test asserting the hyphenated-name case is intentionally NOT stripped (so the next reader knows the strip's blind spot is by design, not by oversight). Surface the choice in 46.1's UAT replay before closing the parent.

### WR-02: `stripLeadingGreeting` matches ANY single word, eliding generic salutations the admin may have intended

**File:** `src/emails/sendUpdate/Body.tsx:23`, `src/emails/sendUpdate/Body.test.ts:46-69`
**Issue:** The regex doesn't validate that the matched word equals `clientFirstName`. So `"Hi everyone,\n\nbody"`, `"Hi all,\n\nbody"`, `"Hi team,\n\nbody"` all get the leading line silently elided when the structural Greeting is `"Hi Sarah,"`. The recipient sees `"Hi Sarah,"` followed by the body — the admin's intentional "Hi all," opener is gone with no log, no error, no warning.

The negative-case test on line 66-69 catches the multi-word `"Hi everyone, this is the body"` but only because there's no trailing newline. Once admin adds the natural blank-line break after their salutation, the strip fires.

This is technically defensible (D-1 says "narrow by design — only the doubled-greeting case"), but it's a content-elision risk: the admin can lose intentional opening text without feedback. The test suite does not cover this scenario at all.

**Fix:** Add a defensive test asserting the behavior is intentional (so future readers see it):
```ts
it("DOES strip 'Hi all,' even when greetingName != 'all' (known false-positive, narrow-by-design)", () => {
  expect(stripLeadingGreeting("Hi all,\n\nBody.")).toBe("Body.");
});
```
Or — and this is the safer functional fix — pass `clientFirstName` to `stripLeadingGreeting` and only strip when the captured word equals it (case-insensitive). That tightens the strip to the exact double-greeting case CONTEXT.md actually describes.

### WR-04: `[data-ogsc] * { color: inherit !important; }` catch-all is broader than its surrounding scoped rules and risks future regressions

**File:** `src/emails/shell/EmailShell.tsx:119`
**Issue:** The universal-selector rule applies to every descendant element under any ancestor with `data-ogsc`. The comment on line 115-118 frames it as "catch-all auto-darken cancellation". The risk: any future section component that relies on a literal inline `style="color:..."` to render a non-inherited color (e.g., a status callout, a warning row) will silently lose its color in Outlook desktop because `* { color: inherit !important }` outranks the inline color via `!important`.

Per CSS specificity rules, an ID selector or higher-specificity class would beat this, but inline style with `!important` would not (because `[data-ogsc] *` plus `!important` matches inline `!important` and resolves cascade by source order; the head-style block is parsed before the body, so head wins on equal-specificity ties when both are `!important` — actually, inline styles have higher specificity than universal even with `!important`, so inline `!important` wins. This is enough of a corner case that it's not obvious to a future developer reading the CSS.).

The narrower fix is to scope the rule to the children that actually need it, not all descendants. Today, the per-class rules above already cover the known palette; the catch-all is a belt-and-suspenders that may overshoot.

**Fix:** Remove the universal-selector rule and trust the scoped rules above; if a future section component adds inline color, lock it via a new `[data-ogsc] .new-class` rule and re-run the merge-gate test in Outlook desktop. If the universal rule is genuinely load-bearing for inline-styled text the planner observed in UAT, leave it in but add a comment naming the specific element it was inserted to protect, so future readers know what regression to watch for.

### WR-05: MSO conditional comment is wrapped in a `<div>` inside `<head>` — invalid HTML, fragile across react-email versions

**File:** `src/emails/shell/EmailShell.tsx:138-143`
**Issue:** The conditional Outlook comment is injected via `<div style={{display:none}} dangerouslySetInnerHTML={...}/>` placed inside `<Head>`. Per HTML5 (and HTML4 before it), `<div>` is not a valid child of `<head>`; only metadata content (title, base, link, meta, noscript, script, style, template) is allowed. The browser's HTML parser will move the `<div>` to `<body>` during parsing, which means the conditional comment may end up in the wrong place once a webmail client re-parses the rendered HTML.

The current rendered output happens to keep the `<div>` in head because react-email's `render()` uses a non-spec-conformant string serializer (it doesn't fix-up parse errors), so the bug is latent — it works today, it could stop working on the next react-email version bump.

**Fix:** Inline the MSO conditional inside the existing `<style>` element on line 82-133. The MSO conditional comment can live above or below the existing CSS rules in the same `<style dangerouslySetInnerHTML>` payload — Outlook desktop processes `<!--[if mso]>` boundaries inside `<style>` tags in older builds. Verify the rendered HTML still contains the conditional in the merge-gate UAT replay. Alternatively, inject the conditional as the second `<style>` element in the head with its CSS payload guarded:
```tsx
<style dangerouslySetInnerHTML={{
  __html: `<!--[if mso]>body { mso-color-scheme: light !important; }<![endif]-->`
}} />
```
Test that EmailShell.test.ts's assertions on `<!--[if mso]>` and `<![endif]-->` still pass (both substring-only).

### IN-05: `Body.tsx` regex `\s*\n+` is fragile due to overlap between `\s` and `\n`

**File:** `src/emails/sendUpdate/Body.tsx:23`
**Issue:** `\s*\n+` is technically valid because regex backtracks, but `\s` includes `\n`, so the engine has to try-and-fail combinations. **Fix:**
```ts

```

## Round 3 Critical Issues

### CR-R3-01: Dark-mode lock palette is dead — `pill-*`, `cta-*`, `text-*`, `border-cream-dark` selectors match zero rendered elements

**File:** `src/emails/shell/EmailShell.tsx:88-130, 142-169`
**Issue:** The Phase 46.1 D-6 (gap-4) `<style>` block and the new D-10/D-11 (gap-7) `@media (prefers-color-scheme: dark)` block both target class hooks that are never applied to any rendered element:

1. **Procurement pill classes** (`pill-scheduled`, `pill-warehouse`, `pill-in-transit`, `pill-ordered`, `pill-pending`, `pill-delivered`, `pill-installed`): `Procurement.tsx:82-96` `pillStyle()` returns inline-style only — there is NO `className` set on the pill `<span>` (`Procurement.tsx:118`). The pill rules in EmailShell.tsx have nothing to bind to.

2. **CTA classes** (`cta-terracotta`, `cta-gold`): `EmailButton.tsx:39-58` returns `<Button>` with inline `style` only — no `className` anywhere. The CTA rules bind to nothing.

3. **Tailwind utility hooks** (`bg-cream`, `text-stone`, `text-stone-light`, `text-charcoal`, `text-mid`, `text-cream`, `text-cream-ivory`, `border-cream-dark`, `text-dark`): `<Body className="bg-cream font-body">` and similar callsites pass these as className inputs to `<Tailwind>`. But Tailwind's email compiler **inlines these classes into the `style` attribute and removes the className from the rendered HTML**. Confirmed by reading the snapshot at `src/emails/sendUpdate/__snapshots__/SendUpdate.test.ts.snap`: there is no `class="bg-cream"`, no `class="text-stone"`, no `class="border-cream-dark"` — instead `<body style="background-color:#FAF8F5">`, `<p style="color:rgb(138,132,120)...">`, etc. `grep -E 'class="[^"]*(bg-cream|pill-|cta-|text-stone|text-charcoal|text-mid|border-cream-dark|text-cream)' SendUpdate.test.ts.snap` returns zero matches.

   The only Tailwind class that survives compilation in the snapshot is the `class="font-body"` on the body (font-family fallback), and even that does not match any of the dark-mode lock rules.

The result: in Outlook desktop dark mode (`[data-ogsc]`/`[data-ogsb]` ancestor) and in any client honoring `prefers-color-scheme: dark`, the only rule that fires is the `body, .bg-cream` body-element selector for background — because `body` is a literal element selector and matches the `<body>` element directly. Everything else (pill bg, pill text, pill border, CTA bg, all heading/body text colors) falls back to Outlook's auto-darken, which is exactly the contrast inversion Liz reported.

The tests in `EmailShell.test.ts:28-30, 80-99, 141-146` assert that the hex values appear in the rendered HTML. They do appear — inside the `<style>` block declarations themselves (190 occurrences of `pill-* {` rule declarations in the snapshot, zero `class="pill-*"` applications). The tests prove the CSS rules were emitted, not that the rules apply to anything. Reading the regression guards as proof of dark-mode protection is a category error.

The new gap-7 `<table bgcolor="#FAF8F5">` wrapper (line 204-212) and inline `<Body style={{backgroundColor:"#FAF8F5"}}>` (line 193) DO work — they target elements directly via HTML attribute and inline style. So the body background survives. Body content does not.

Additional gap not previously caught: `WorkOrder.tsx:48` uses `text-stone-dark`, but EmailShell.tsx has no `[data-ogsc] .text-stone-dark` rule and the `@media` block doesn't include it either. Even if className hooks were preserved, this Tailwind class would be unprotected.

**Fix:** Two options:

**Option A (smaller change, preserves existing CSS shape):** Apply explicit className strings to the elements that need to survive dark mode, in addition to (or instead of) Tailwind classes. For `Procurement.tsx` pill `<span>`:
```tsx
<span
  className={`pill-${it.status}`}
  style={pillStyle(it.status)}
>
  {STATUS_LABELS[it.status]}
</span>
```
For `EmailButton.tsx`:
```tsx
<Button href={href} className={`cta-${variant}`} style={...}>
```
For body, headings, sub-text, etc.: ensure each Tailwind class that has a corresponding `[data-ogsc]` lock is also written through `className=` in a form that survives the Tailwind compiler — this likely means switching from `text-stone` (Tailwind utility) to a literal class like `class="text-stone"` injected via a non-Tailwind code path, or adding a `data-color-token="stone"` attribute and rewriting the locks as `[data-ogsc] [data-color-token="stone"] { color: #8A8478 !important; }`.

The data-attribute approach is the more robust fix because it's immune to Tailwind compilation behavior across versions.

**Option B (larger change, cleaner outcome):** Drop the class-hook lock entirely and lean on the gap-7 `prefers-color-scheme` block + the inline-style + `<table bgcolor>` paint that already work. Re-run merge-gate UAT in Outlook desktop and Outlook-for-Mac dark mode. ## Round 4 Critical Issues

### CR-R4-01: MSO conditional placed inside `<style>` element does not activate Outlook conditional comments

**File:** `src/emails/shell/EmailShell.tsx:166-170`
**Issue:** The WR-05 fix moved the MSO conditional out of an invalid `<div style="display:none">` wrapper (which violated HTML5's "head can only contain metadata content" rule) into a sibling `<style>` element:

```tsx
<style
  dangerouslySetInnerHTML={{
    __html: `<!--[if mso]>body { mso-color-scheme: light !important; }<![endif]-->`,
  }}
/>
```

The rendered output is `<style><!--[if mso]>body { mso-color-scheme: light !important; }<![endif]--></style>` (verified in `WorkOrder.test.ts.snap:87` and `SendUpdate.test.ts.snap:123`).

**This does not work as an Outlook MSO conditional.** MSO conditional comments are an HTML-level construct: Outlook's HTML parser scans HTML content looking for `<!--[if mso]>...<![endif]-->` boundaries and interprets the wrapped content as HTML to render in MSO context. Inside a `<style>` element, the parser is in "CSS raw text" mode. The CSS tokenizer sees:
1. `<!--` — interpreted as a CSS CDO (Comment Delimiter Open) token, a legacy SGML "hide-from-old-browsers" marker. It is consumed and ignored.
2. `[if mso]>body { mso-color-scheme: light !important; }` — interpreted as CSS. `[if mso]` is parsed as an attribute presence selector (`[if=mso]`); `>body` is the child-combinator + type selector. The full selector is `[if=mso] > body`, which never matches anything (no element in the rendered HTML has an `if` attribute).
3. `<!--` (the second one in `<![endif]-->`) — wait, the actual sequence is `<![endif]-->`. The CSS tokenizer sees `<![endif]` (an unknown construct, parsed as a malformed selector and discarded along with the rest of that rule), and finally `-->` is consumed as a CDC (Comment Delimiter Close) token, ignored.

Net effect: the entire payload is parsed as CSS that matches nothing. The `mso-color-scheme: light !important` directive that was the whole point of the WR-05 carryover never reaches Outlook desktop.

**Three lines of evidence:**
1. The snapshot shows `<style><!--[if mso]>body { mso-color-scheme: light !important; }<![endif]--></style>`. This is byte-identical to what the `dangerouslySetInnerHTML` payload produces (no Outlook activation transforms occur — the `<style>` element prevents HTML-level interpretation of the conditional markers).
2. The CSS Working Group spec (CSS Syntax Module Level 3, §4.3.2 "Consume comments") explicitly defines `<!--` and `-->` as CDO/CDC tokens consumed and ignored at the top level of a CSS block.
3. Outlook's documented MSO conditional comment behavior (Microsoft's "Word Filter HTML" docs, mirrored in the litmus.com test corpus) requires the conditional to appear in HTML context: typically in `<body>`, occasionally directly in `<head>` as a sibling of `<meta>`/`<link>`/`<style>` elements — never *inside* a `<style>` element where CSS parsing rules apply.

The original Phase 46 implementation (per WR-05's diagnosis) had the right idea but used an invalid container (`<div style="display:none">` in `<head>`). The round-4 fix eliminated the invalid container by moving the markup inside a `<style>` element — a valid `<head>` child — but in doing so changed the parsing context from HTML to CSS, breaking the conditional.

**Test surface is misleading:** `EmailShell.test.ts:90-103` asserts that `<!--[if mso]>`, `<![endif]-->`, and `mso-color-scheme: light` substrings appear in the rendered HTML. All three substrings DO appear (in the CSS-context `<style>` body), so the tests pass. But they pass against a non-functional conditional — the same category-error CR-R3-01 documented for the dead pill-class rules. The assertions prove the substrings are emitted, not that Outlook parses them as a conditional.

**Why this is a BLOCKER:** Phase 46.1's gap-4 closure relies on a layered defense: `[data-ogsc]/[data-ogsb]` for newer Outlook builds + `@media (prefers-color-scheme: dark)` for Outlook variants honoring CSS media queries + MSO conditional + `<table bgcolor>` HTML attribute paint + inline body bg. The MSO conditional was the layer specifically targeting **older Outlook desktop builds where `[data-ogs*]` doesn't reach** (per the comment on line 161). Without it, those older builds are unprotected. Liz's UAT covers Outlook desktop dark mode broadly; if her test environment includes any Outlook 2016/2019 build that doesn't honor `[data-ogs*]`, the gap-4 regression Liz reported will resurface.

The `<table bgcolor>` and inline `<body style="background-color">` paints DO survive into older Outlook builds (HTML attributes are interpreted natively, no CSS parser involved), so the body-bg layer remains intact. But `mso-color-scheme: light !important` was the directive that locked the page-level color scheme for Outlook desktop's word-engine renderer — without it, the word-engine renders its default (which respects user dark-mode preference in modern Outlook).

**Fix:** Two options.

**Option A — Move the conditional back to HTML context, but use a valid container.** Place the conditional as a child of `<Body>` (after `<Preheader>`, before the `<table bgcolor>` wrapper). Since `<Body>` is the rendered `<body>` element, the conditional sits in HTML context where Outlook's parser interprets it:

```tsx
<Body className="bg-cream font-body" style={{ backgroundColor: "#FAF8F5" }}>
  <Preheader>{preheader}</Preheader>
  {/* MSO conditional -- in HTML context so Outlook parses it. */}
  <div dangerouslySetInnerHTML={{
    __html: `<!--[if mso]><style>body { mso-color-scheme: light !important; }</style><![endif]-->`
  }} />
  <table bgcolor="#FAF8F5" ...>
```

Note the conditional now wraps a `<style>` block — Outlook MSO renders `<style>` content inside conditional comments because the conditional's payload is parsed as HTML.

**Option B — Inline the directive in the existing main `<style>` element via `@media` query that targets MSO.** Outlook desktop interprets `mso-` prefixed properties anywhere they appear in CSS, regardless of media queries. So:

```css
@media all {
  body { mso-color-scheme: light !important; }
}
```

This drops the conditional comment entirely. The `mso-color-scheme` directive sits as a plain CSS rule that Outlook desktop honors. Other CSS engines (webmail clients) ignore unknown `mso-*` properties per CSS spec. Verify in the merge-gate UAT replay that Outlook 2016/2019 desktop respects this.

**Option C — Use a sibling `<comment>` or text-node MSO conditional.** React doesn't expose a native way to emit raw HTML comments at the top level of `<head>`, but a `<style media="all and (mso-condition: light)">...</style>` style attribute could carry the same intent — though `mso-condition` is non-standard and untested in the merge-gate environment.

**Recommendation:** Option A (move the conditional to `<Body>` so it sits in HTML context). This requires no new CSS or non-standard properties; it just relocates the existing payload. Verify in the UAT replay that the snapshot tests still pass (their substring assertions on `<!--[if mso]>` and `<![endif]-->` are location-agnostic, so they will).

**Verification:** After fixing, tighten the EmailShell.test.ts assertions to specifically lock the conditional in HTML context, e.g., `expect(html).toMatch(/<body\b[^>]*>[\s\S]*?<!--\[if mso\]>/)` — proves the conditional sits in the document body, not inside a `<style>` element. (And the `\b` boundary prevents the `<bodyfoo>` false-positive that round-3 IN-R3-03 documented for the body-bg test.)

### WR-R4-02: `SendUpdate.tsx:173` `clientFirstName ?? ""` coercion silently disables the strip when caller doesn't supply firstName

**File:** `src/emails/sendUpdate/SendUpdate.tsx:173`
**Issue:** SendUpdate calls `<Body personalNote={personalNote} clientFirstName={clientFirstName ?? ""} />`. The `?? Then `stripLeadingGreeting("Hi Victoria,\n\nBody", "")` is called. The new tightened regex (Body.tsx:35-44) requires `capturedFirstName.toLowerCase() === clientFirstName.toLowerCase()`, which becomes `"victoria" === ""` → false → no strip.

Net effect: when `clientFirstName` is undefined upstream, the gap-1 double-greeting fix silently does nothing. Admin types `"Hi Victoria,\n\nBody"`, Greeting renders its fallback (likely `"Hi there,"` or similar — Greeting.tsx not in this round's scope), and the Body renders `"Hi Victoria,..."` unstripped. Two greetings ship.

This is a regression of the gap-1 closure for any code path that doesn't supply `clientFirstName`. The Phase 46.1 production call site (the `/api/send-update` POST handler) presumably reads `clientFirstName` from the Sanity client document and passes it through, so the production hot path is OK. But:
- Test code that uses `baseInput()` without overriding `clientFirstName` — `Body.test.ts:139` does set it, so that's covered. Other call sites need audit.
- Future code paths (e.g., a "preview without selecting a client" admin tool) may not have a firstName.
- Sanity client documents missing the firstName field (data quality issue) bypass the strip.

The original round-3 contract for the strip was "narrow by design — only the doubled-greeting case." The round-4 implementation upholds that narrowness, but the `?? ""` coercion at the SendUpdate boundary silently widens the constraint — now the strip is "narrow by design AND only when caller supplies firstName."

**Fix:** Two options.

**Option A — Make the type contract enforce the requirement.** Change `SendUpdateEmailInput.clientFirstName?: string` (line 67) to `clientFirstName: string`, dropping the `?`. Audit all callers; update fixtures and tests to always pass firstName (default to a placeholder like `"there"` if no real value exists). **Option B — Add a Body-level guard.** When `clientFirstName === ""`, fall back to a more permissive strip (e.g., the original `\w+` regex without the exact-match check) so admin's intentional `"Hi Victoria,..."` is still stripped even when firstName is unknown. This widens the WR-02 content-elision risk for the unknown-firstName case, but the alternative (silent double-greeting) is worse for the user.

**Option C — Document the constraint.** Add a comment in `SendUpdate.tsx:173` that explains why the `?? ""` coercion exists and what the trade-off is. Add a test in `Body.test.ts` asserting that `stripLeadingGreeting("Hi Victoria,\n\nBody", "")` does NOT strip (so future readers see the intentional limitation).

**Recommendation:** Option C as a minimum (cheapest, no behavior change). Option A is the cleanest long-term fix but requires upstream audit of all call sites.
