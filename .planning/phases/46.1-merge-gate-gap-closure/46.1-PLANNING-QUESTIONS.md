## 2. GSD workflow primer (for someone new)

This project uses the **GSD (Get Shit Done)** workflow — a structured `~/.planning/` directory with these conventions:

| Artifact | Purpose |
|----------|---------|
| `STATE.md` | Project status, current position, next action |
| `ROADMAP.md` | Phase list with goals + status |
| `REQUIREMENTS.md` | Numbered requirement IDs (REQ-01, REQ-02, …) |
| `phases/{N}-{slug}/` | Per-phase directory |
| `{N}-CONTEXT.md` | Locked decisions for the phase (D-1, D-2, …) |
| `{N}-PLAN.md` (or `{N}-NN-PLAN.md` for multi-plan waves) | Executable task list with frontmatter |
| `{N}-SUMMARY.md` | What was built, written after execution |
| `{N}-UAT.md` | Human acceptance test results |
| `{N}-VERIFICATION.md` | Goal-backward verification — does the code actually deliver the phase goal? |
| `{N}-HUMAN-UAT.md` | Mirror of UAT for human-only visual gates |

**Decimal phases (e.g., 46.1)** are **gap-closure phases** — small follow-ups to a parent phase that exists to fix UAT findings without re-opening the parent.

**"Gap-N"** = a specific UAT finding. E.g., gap-1 = greeting double-render. Each gap has a `status: failed | resolved`.

**"D-N"** = a locked design decision in CONTEXT.md. E.g., 46.1-CONTEXT.md D-7 = "all table cell content left-aligned." Decisions are immutable once locked.

**"REQ-N"** = a numbered project requirement from REQUIREMENTS.md. Gap-closure phases like 46.1 don't carry REQ-IDs — they carry `parent_uat_gap` references instead.

**Plan-phase commands:**
- `/gsd-plan-phase {N}` — fresh planning
- `/gsd-plan-phase {N} --gaps` — produce additional plans to close gaps from a verification or UAT report (this is what I'm running now)

After planning: `/gsd-execute-phase {N}` runs the plans (one atomic commit per plan).

## 5. The two new gaps in detail

### gap-6 — Procurement table layout

**File:** `src/emails/sendUpdate/Procurement.tsx`

**What's wrong:**
- Item column has no explicit width → react-email distributes width by content-fit → Item column gets squeezed when items are long.
- Three rows in Liz's test render: Row 1 (Custom Sectional Sofa — Kravet Crypton Fabric) wraps to 2 lines; Row 2 (Handknotted Wool & Silk Rug — 9x12) fits on 1 line; Row 3 (Italian Pendant Lights set of 3 — Flos IC) wraps to 2 lines.
- `<Column verticalAlign="middle">` on each cell anchors content to row vertical center. With 2-line rows interleaved with 1-line rows, ETA cells sit at three different y-positions — a zig-zag, not a stripe.
- Procurement section's outer spacing reads tight: `paddingTop: 10` (Milestones is 12, ReviewItems is 12 + outer Section paddingTop 8). Liz wants more separation.

**Fix shape (UAT recommendation):**
1. Explicit column widths — Item ~60%, Status ~22%, ETA ~18% (in 600px email card)
2. `verticalAlign="top"` on row Columns so ETA stripe stays clean even when Item wraps
3. Increase marginTop above Procurement Section

**Tests:** snapshot regen + new section-scoped HTML inspection asserting `<col>` / `<td width=...>` present on Procurement columns, Item column has the largest declared width, verticalAlign value matches the locked decision. Visual confirmation requires round-4 re-test.

### gap-7 — Outlook for Mac dark-mode persists

**File:** `src/emails/shell/EmailShell.tsx`

**What's wrong:** The 46.1-04 fix (`[data-ogsc]` text-color rules, `[data-ogsb]` bg rules, MSO conditional) targets **classic Outlook for Windows**'s auto-darken pipeline. **New Outlook for Mac** appears to use a different dark-mode pipeline that either doesn't paint the `data-ogsc`/`data-ogsb` attributes onto inline-styled elements (so the rules never match), or applies its dark-mode CSS at higher specificity than `!important`, or uses a different selector hook altogether.

**Why this matters technically:** Liz uses Outlook for Mac as her daily driver, so the failing variant *is* her actual rendering surface. We can't ship "works on Windows desktop" if Mac is where she actually reads.

**Fix candidates (UAT-recommended; planner to research/spike):**
1. **Force background paint at body/wrapper level** — instead of relying on inline styles being preserved, paint cream `#FAF8F5` background via a top-level `<table bgcolor="#FAF8F5">` or `<div style="background:#FAF8F5">` that Outlook-Mac cannot easily override. This is the "Outlook Mac dark-mode survival" pattern documented in HTML-email community sources.
2. **Add `@media (prefers-color-scheme: dark)`** with explicit overrides matching the locked palette. Some Outlook variants honor this even when ignoring `color-scheme: only light`.
3. **Add `class~="email-shell"` selectors with `!important`** as a third selector path that may match where `[data-ogsc]` doesn't.
4. **Investigate** — does New-Outlook-for-Mac strip `<style>` blocks in `<head>`? (If yes, all our locks are dead before they execute and we need inline-attribute paint only.) Does body-level `style="background:#X"` survive as an inline attribute when class-level CSS doesn't?

**Tests:** depends on chosen path. Likely HTML-inspection assertions for marker presence + round-4 visual gate.
