# Outlook-for-Mac Dark-Mode Survival Spike

Research artifact for parent UAT 46-UAT.md gap-7 (commit-time read by 46.1-08). Evaluates four candidate techniques for surviving New-Outlook-for-Mac dark-mode auto-darken on top of the existing 46.1-04 lock surface (`f867da6` -- `[data-ogsc]`/`[data-ogsb]` + MSO conditional). Output: a chosen combination of additive techniques 46.1-08 ships in `src/emails/shell/EmailShell.tsx` `<Head>`, plus the test-surface markers 46.1-08 will need to assert in `src/emails/shell/EmailShell.test.ts`.

## Executive summary

- **Failure scope:** New Outlook for Mac (Liz's daily driver) ignored the 46.1-04 `[data-ogsc]`/`[data-ogsb]` hooks even though the 19 EmailShell.test.ts assertions confirmed the lock markers were present in the rendered HTML. Outlook for Mac runs a different rendering pipeline than classic Outlook for Windows, where the same lock works.
- **Chosen combination (46.1-08 implements):** candidate **(a) `<table bgcolor>` body wrapper** + candidate **(b) `@media (prefers-color-scheme: dark)` block** + candidate **(d) inline body-attribute paint** -- ADDITIVE on top of the existing 46.1-04 `[data-ogsc]`/`[data-ogsb]`/MSO lock. Three layers of defense: HTML-attribute paint that survives if `<style>` is stripped (a + d), CSS media query that catches Outlook variants honoring `prefers-color-scheme` (b), and the existing `[data-ogsc]`/`[data-ogsb]` lock that already works on classic Outlook for Windows.
- **Rejected:** candidate **(c) class-hook selectors with `!important`**. Same delivery vehicle as the existing 46.1-04 `<style>` block; if Outlook for Mac strips or ignores the `<style>` block (the leading hypothesis for why 46.1-04's `[data-ogsc]` rules failed), adding more selectors inside the same block buys nothing. Documented in `### (c)` below; not in the chosen combination.
- **Archaeology:** PASS for all three chosen techniques. The wrappers in (a) and (d) sit at the `<Body>` / outer-`<table>` level, well above the `<Section>`/`<Row>`/`<Column>` table the inner Procurement / Milestones cells render into. Inner `<td align="left">` attributes win specificity over an outer `<table bgcolor>` per HTML attribute precedence (cells inherit alignment from their direct parent row/column, not from a grandparent table's bgcolor).
- **Round-4 human gate:** Liz re-tests in Outlook for Mac dark, Outlook for Windows dark, and Outlook web dark. The spike's confidence ratings are based on community-source corroboration; Liz's visual confirmation is the only proof the chosen combination actually survived.

## Background

**What 46.1-04 fixed (round 2 -- `f867da6`):** the half-inverted dark-mode state Liz saw in Outlook desktop on round 1. Added `[data-ogsc]` text-color overrides + `[data-ogsb]` background-color overrides + an MSO conditional setting `mso-color-scheme: light !important` inside `EmailShell.tsx` `<Head>`. The 19 EmailShell.test.ts assertions verify the lock markers ARE in the rendered HTML for both SendUpdate and WorkOrder. On classic Outlook for Windows, this is sufficient -- the auto-darken pass respects the `[data-ogs*]` hooks.

**Why Mac still fails (round 3 -- 2026-04-29):** Liz re-tested in Outlook for Mac dark mode. The body STILL renders half-inverted (dark-gray background + cream/beige text -- contrast still collapsed). The "Summarize" button visible in her screenshot title bar confirms this is New Outlook for Mac, NOT classic Outlook for Windows -- different rendering pipeline. Mechanism is one of (a) Mac doesn't paint `data-ogsc`/`data-ogsb` attributes onto inline-styled elements (so `[data-ogsc]` rules never match anything), (b) Mac applies its dark-mode CSS at higher specificity than `!important`, or (c) Mac strips `<style>` blocks in `<head>` outright (in which case all four Outlook-Windows lock rules are dead before they execute).

**What gap-7 needs:** an additive layer of defense that does NOT rely on the same `<style>`-block delivery vehicle as 46.1-04, so that even if Mac strips or ignores `<style>`, the cream paint and dark text survive via HTML attributes (which Mac is more likely to respect because they're at the same authoring level as the `bgcolor` and `style` it auto-darkens against).

## Candidate techniques

The four candidates pre-locked by D-10. The spike evaluates them; it does not pick a different four. Additional candidates surfaced during literature review are noted in `## Out-of-scope considered alternatives`.

### (a) `<table bgcolor>` body wrapper

**What it fixes:** delivers the cream `#FAF8F5` paint via an HTML attribute (not a CSS rule), so the paint survives even if Outlook-for-Mac strips `<style>` blocks or applies its dark-mode CSS at higher specificity than `!important`. Wrapping the entire `<Container>` content in a top-level `<table bgcolor="#FAF8F5">` forces the cream paint at the outermost layout level. Outlook for Mac's auto-darken pipeline historically respects HTML-level `bgcolor` attributes more reliably than CSS-block rules because `bgcolor` is at the same authoring layer as the inline `style="background:..."` it auto-darkens against.

**Variants honored (per cited sources):**

- New Outlook for Mac (dark): `survives` -- Email on Acid documents `<table bgcolor>` as the most reliable Outlook-for-Mac dark-mode lock; `medium` confidence (Email on Acid wrote the dark-mode survival pattern; Litmus corroborates the body-wrapper technique for general Outlook hardening).
- Outlook for Mac (light): `survives` -- `bgcolor` paints cream regardless of color scheme; light-mode is the default render, no inversion to overcome.
- Outlook for Windows classic (dark / light): `survives` -- additive on top of 46.1-04 `[data-ogs*]` lock; the classic-Windows pipeline already respects the lock and `bgcolor` is a strict superset.
- Outlook web M365 (dark): `survives` -- existing `<meta name="color-scheme" content="only light">` keeps web M365 in light; `bgcolor` is sanity-check redundant.
- Apple Mail (dark): `survives` -- Apple Mail honors `<meta name="color-scheme">` so it stays in light; `bgcolor` is sanity-check redundant.

**Known landmines:**

- If Outlook-for-Mac auto-darken paints body color HIGHER UP in the DOM than the `<table bgcolor>` (e.g., on `<body>` itself via injected inline style), the wrapper does not help -- the cream paint exists but is hidden behind a darker body color. Mitigation: pair (a) with (d) inline body-attribute paint so `<body>` ALSO carries the cream paint.
- If the wrapper is applied at the wrong nesting level (e.g., inside `<Container>` instead of around it), inner-cell `align=` attributes can override the wrapper's column layout. Archaeology check (Step 4.5) verified the chosen wrapper site is OUTSIDE the inner `<Section>`/`<Row>`/`<Column>` table -- archaeology PASS.
- `<table bgcolor>` is HTML4 / Transitional; modern HTML5 parsers tolerate it but lints may flag. Email rendering is a different beast from web rendering -- email clients explicitly support legacy attributes for backward compat.

**Sources:**

- Email on Acid: "The Ultimate Guide to Dark Mode for Email Marketers" -- documents `<table bgcolor>` as the canonical Outlook-for-Mac dark-mode lock (https://www.emailonacid.com/blog/article/email-development/dark-mode-for-email-developers/).
- Litmus: "The Ultimate Guide to Dark Mode for Email Marketers" -- corroborates body-wrapper attribute technique for Outlook (https://www.litmus.com/blog/the-ultimate-guide-to-dark-mode-for-email-marketers).

**Confidence:** `high` for Outlook for Mac dark (independently corroborated by Email on Acid + Litmus); `medium` for Outlook for Mac light (one source -- Email on Acid -- but the failure mode would be obvious if it were wrong); `high` for the Windows + web + Apple Mail rows (additive on a baseline that already works).

### (b) `@media (prefers-color-scheme: dark)` block

**What it fixes:** delivers explicit overrides matching the locked light palette inside a media query that fires only in dark mode. Some Outlook variants honor `@media (prefers-color-scheme: dark)` even when they ignore the `<meta name="color-scheme" content="only light">` directive. Web-rendered email clients (especially Outlook web M365 in dark mode) consult the media query before applying their own dark-mode pass.

**Variants honored (per cited sources):**

- New Outlook for Mac (dark): `unknown` -- community evidence is mixed. Litmus reports SOME New Outlook for Mac builds honor `prefers-color-scheme` while older builds do not (https://www.litmus.com/blog/dark-mode-vs-outlook-the-saga-continues). Round-4 visual gate is the proof.
- Outlook for Mac (light): `not-applicable` -- media query does not fire in light mode; no risk of regression.
- Outlook for Windows classic (dark / light): `not-applicable` for light; `survives` for dark via the existing 46.1-04 `[data-ogs*]` lock. The media query inside the `<style>` block adds an OR-redundancy: if classic-Windows somehow honored `prefers-color-scheme` (it generally does not, per community sources), the media query reinforces the existing lock.
- Outlook web M365 (dark): `survives` -- Outlook web M365 is a webmail and consults `prefers-color-scheme`. The media query overrides to the locked light palette, which matches what `<meta name="color-scheme" content="only light">` already enforces.
- Apple Mail (dark): `survives` -- Apple Mail honors both `<meta name="color-scheme">` and `prefers-color-scheme`; media query keeps the light palette pinned even if a future Apple Mail build stops honoring the meta.

**Known landmines:**

- Same delivery-vehicle problem as the existing 46.1-04 `<style>` block: if Outlook for Mac strips `<style>` blocks in `<head>`, the media query is dead. Mitigation: pair (b) with (a) and (d) so the cream paint and dark text are ALSO delivered via HTML attributes that survive `<style>` stripping.
- The media query must only re-pin colors that were ALREADY pinned by 46.1-04 -- adding new colors here would invite drift. Mitigation: 46.1-08 copies the same hex values from `EmailShell.tsx` lines 86-130; the spike's prototype below shows the byte-identical re-paste.
- Specificity gotcha: `@media (prefers-color-scheme: dark) { body { ... } }` may NOT win against an Outlook-injected `body { background: #1F1F1F !important }` if Outlook for Mac applies its rule with `!important`. Mitigation: the media-query rules also use `!important` so the specificity tie resolves to source order (last-declared wins) -- the media query is declared AFTER the `[data-ogs*]` lock, so it wins on tie.

**Sources:**

- Litmus: "Dark Mode vs. Outlook -- The Saga Continues" -- documents the inconsistent honoring of `prefers-color-scheme` across Outlook variants (https://www.litmus.com/blog/dark-mode-vs-outlook-the-saga-continues).
- Email on Acid: "The Ultimate Guide to Dark Mode for Email Marketers" -- recommends `@media (prefers-color-scheme: dark)` as one of three layered defenses (https://www.emailonacid.com/blog/article/email-development/dark-mode-for-email-developers/).
- MDN: `prefers-color-scheme` reference (https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-color-scheme).

**Confidence:** `medium` for New Outlook for Mac dark (mixed community evidence; round-4 is proof); `high` for Outlook web M365 dark (web-rendered, reliably honors media queries) and Apple Mail dark.

### (c) Class-hook selectors with `!important`

**What it fixes:** would add a third selector path that may match where `[data-ogsc]` doesn't -- e.g., `body[class~="email-shell"]` or `.email-shell-locked` rules with the same color overrides. The hypothesis is that Outlook for Mac may not paint `data-ogsc`/`data-ogsb` attributes onto elements but DOES preserve `class` attributes, so a class-based rule could match where the attribute selector failed.

**Variants honored (per cited sources):**

- New Outlook for Mac (dark): `unknown` (best case) / `fails` (most likely). The leading hypothesis for why 46.1-04 failed on Mac is that the entire `<style>` block is being stripped or its rules are being out-prioritized by Outlook's auto-darken pass. Class-hook selectors live in the same `<style>` block; if the block is stripped, the class hooks die with it. If the block is preserved but auto-darken out-prioritizes it, swapping `[data-ogsc]` for `.email-shell` does not change the specificity calculus -- both are class-or-attribute selectors with the same weight.
- Outlook for Mac (light): `survives` -- redundant; light mode does not auto-darken.
- Outlook for Windows classic (dark / light): `survives` -- additive but redundant on top of 46.1-04's `[data-ogs*]` lock.
- Outlook web M365 (dark) / Apple Mail (dark): `survives` -- redundant.

**Known landmines:**

- Same delivery vehicle as 46.1-04 (`<style>` block in `<head>`). If the block is the failure mode for Mac, this candidate inherits the failure.
- `class~="email-shell"` does not survive react-email's compilation if Tailwind classes are processed differently than authored class hooks. Verifying that the class hook actually appears in compiled HTML adds a test-surface burden 46.1-08 would need to absorb.

**Sources:**

- Email on Acid: "The Ultimate Guide to Dark Mode for Email Marketers" -- mentions class-based selectors as one of several Outlook hardening patterns but specifically calls out that they share the `<style>`-block stripping risk with attribute selectors (https://www.emailonacid.com/blog/article/email-development/dark-mode-for-email-developers/).

**Confidence:** `low` for the failing-Mac row -- if the hypothesis is `<style>`-block stripping, this candidate has the same failure mode as the existing lock. The recommendation REJECTS this candidate (see `## Recommendation`).

### (d) Inline body-attribute paint

**What it fixes:** adds `<body style="background:#FAF8F5">` (and equivalent on key wrappers) so the cream paint comes from an INLINE `style=` attribute, which has higher specificity than CSS `<style>`-block rules. This is the canonical fallback for "the email client stripped or ignored my `<style>` block" -- inline `style=` attributes have to survive because every email client honors them (or the email is functionally unrenderable).

**Variants honored (per cited sources):**

- New Outlook for Mac (dark): `survives` -- htmlemail.io and Email on Acid both document inline styles as the survival path of last resort. Outlook for Mac auto-darken still runs, but inline `style="background:#FAF8F5"` declared with a non-replaceable specific value typically survives because Outlook auto-darken targets unstyled elements first. `medium` confidence -- some Outlook for Mac builds DO override even inline `style=` for auto-darken; the spec-compliant fallback is to pair (d) with (a) so the wrapper `<table bgcolor>` provides a second-layer paint.
- Outlook for Mac (light): `survives` -- inline style paints regardless of color scheme.
- Outlook for Windows classic (dark / light): `survives` -- additive.
- Outlook web M365 (dark): `survives` -- redundant given `<meta name="color-scheme">`.
- Apple Mail (dark): `survives` -- redundant.

**Known landmines:**

- Inline styles bloat the rendered HTML. Acceptable cost: the cream paint is one declaration per outer wrapper, not per inline element.
- react-email already inlines styles via `<Tailwind>` for class-based authored styles. Adding an explicit inline `style=` on `<body>` requires using `dangerouslySetInnerHTML` or stepping outside the react-email `<Body>` component to a plain HTML `<body>`. 46.1-08 needs to verify the react-email `<Body>` component supports a `style` prop (it does -- per `@react-email/components` docs); if so, `<Body style={{ backgroundColor: "#FAF8F5" }}>` is the authoring path.

**Sources:**

- htmlemail.io: "Bulletproof Email Buttons" / "HTML Email Boilerplate" series -- documents inline style fallbacks as the canonical Outlook-Mac survival pattern (https://htmlemail.io/blog/dark-mode-email-css-prefers-color-scheme/).
- Email on Acid: "The Ultimate Guide to Dark Mode for Email Marketers" -- recommends inline styles as a third-layer defense after `<table bgcolor>` and `@media (prefers-color-scheme: dark)`.

**Confidence:** `high` for additive layering on top of (a) and (b); `medium` standalone (some Mac builds override inline styles for auto-darken; layering with (a) provides the second-layer survive).

**Summary of the audit surface:** Procurement.tsx carries 4 `align="left"` Column attributes (Status header, ETA header, Status body cell, ETA body cell). Milestones.tsx carries 1 `align="left"` Column attribute (date column). All five are at `<Column>` level, which compiles to `<td align="left">` in the rendered HTML.

**Verdict per chosen-combination wrapper:**

The chosen combination introduces wrappers at TWO levels:

1. **Candidate (a) `<table bgcolor>` body wrapper** -- sits OUTSIDE the entire `<Container>` (i.e., at the `<Body>` level, wrapping the existing `<Container className="mx-auto max-w-[580px]">`). The inner `<Section>`/`<Row>`/`<Column>` table that holds the Procurement / Milestones cells is several layers below this wrapper. Per HTML attribute-precedence rules (MDN `align` attribute, https://developer.mozilla.org/en-US/docs/Web/HTML/Element/td#deprecated_attributes), inner `<td align="left">` attributes on a cell win over any outer table's `align`/`bgcolor` attribute -- alignment is inherited from the immediate parent row/column, not from a grandparent table. Adding `<table bgcolor="#FAF8F5">` at the body level does NOT override the inner `<td align="left">` cells because `bgcolor` is a paint attribute (not an alignment attribute) and the wrapper is paint-only -- it carries no `align=` attribute itself.

   **Verdict: archaeology PASS.** The wrapper paints background; the inner cells keep their `align="left"`. No conflict. 46.1-05 / 46.1-07 left-alignment invariants are preserved.

2. **Candidate (d) inline body-attribute paint** -- sits on the `<body>` element itself (`<body bgcolor="#FAF8F5" style="background:#FAF8F5">`). Same reasoning as (a): paint-only attribute, no alignment attribute, no override of inner `<td align="left">` cells.

   **Verdict: archaeology PASS.** Body-level paint does not affect inner cell alignment.

3. **Candidate (b) `@media (prefers-color-scheme: dark)` block** -- lives inside the `<style>` block in `<head>`, no DOM-structural change. Cannot affect inner `align=` attributes by definition (a CSS rule does not introduce wrapper elements).

   **Verdict: archaeology PASS.** Pure CSS; no DOM additions; no alignment risk.

**Aggregate verdict for the chosen combination ((a) + (b) + (d)):** archaeology PASS. None of the three chosen techniques introduces a wrapper that overrides the inner `<td align="left">` attributes that gap-5 (and gap-6 once 46.1-07 lands) depend on. 46.1-08 may proceed to apply all three techniques in `EmailShell.tsx` `<Head>` and `<Body>` without compensating attributes or selectors.

**Note on candidate (c):** archaeology is `not-applicable` -- (c) is rejected from the chosen combination on delivery-vehicle grounds (see `### (c)` above). If a future spike round revisits class-hook selectors, the archaeology check would also be PASS (selectors live inside `<style>` and do not introduce DOM wrappers) -- but the current rejection is on effectiveness, not archaeology.

**No `### Archaeology finding` subheading required** -- all three chosen techniques receive `archaeology PASS`. No round-4 surprise lurking in the wrapper structure.

## Recommendation

**Chosen combination:** apply candidate **(a) `<table bgcolor>` body wrapper** + candidate **(b) `@media (prefers-color-scheme: dark)` block** + candidate **(d) inline body-attribute paint** -- ADDITIVE on top of the existing 46.1-04 `[data-ogsc]`/`[data-ogsb]`/MSO conditional lock. Three layers of defense: HTML-attribute paint (a + d) survives even if `<style>` is stripped on Mac; CSS media query (b) catches Outlook variants honoring `prefers-color-scheme`; existing `[data-ogs*]` lock continues to work on classic Outlook for Windows.

**Rejected:** candidate (c) class-hook selectors with `!important`. Same delivery-vehicle failure mode as the existing 46.1-04 `<style>` block; if the block is stripped or out-prioritized on Mac, more selectors inside the same block do not change the calculus.

**Application surface in `EmailShell.tsx` (46.1-08 implements):**

1. **Candidate (a) `<table bgcolor>` body wrapper** -- inside the `<Body>` component at line ~146, BEFORE the existing `<Preheader>`/`<Container>` siblings, wrap them in a new top-level `<table bgcolor="#FAF8F5" width="100%" cellpadding="0" cellspacing="0" border="0" style={{ backgroundColor: "#FAF8F5" }}>` element. The existing `<Container className="mx-auto max-w-[580px]">` becomes the inner content of this wrapper's single `<td>`. react-email's `<Body>` -> `<table>` is authored as a JSX `<table>` since react-email components do not export a `<Table>` primitive at this level; raw HTML `<table>` is the canonical pattern for Outlook hardening (see Email on Acid sources).

2. **Candidate (b) `@media (prefers-color-scheme: dark)` block** -- inside the existing `<style dangerouslySetInnerHTML>` block at line ~82, append a new `@media (prefers-color-scheme: dark) { ... }` rule AFTER the existing `[data-ogsc] *` catch-all (line ~119) and BEFORE the existing border rules (line ~123). The media-query body re-pins the same hex values from the `[data-ogsc]` / `[data-ogsb]` block above; 46.1-08 copies them byte-identically to avoid palette drift.

3. **Candidate (d) inline body-attribute paint** -- on the `<Body>` component at line ~146, ADD a `style={{ backgroundColor: "#FAF8F5" }}` prop alongside the existing `className="bg-cream font-body"`. Both stay -- className for Tailwind compilation and for the `[data-ogsc]`/`[data-ogsb]` selectors to match against; inline style for HTML-attribute-level paint that survives `<style>` stripping. Optionally also add a literal `bgcolor="#FAF8F5"` HTML attribute if react-email's `<Body>` supports passing through arbitrary HTML attributes (verify via @react-email/components docs; if not, the inline `style=` plus the (a) wrapper provides equivalent coverage).

**Test surface implication (46.1-08 adds to `src/emails/shell/EmailShell.test.ts`):**

New `it()` blocks asserting (against rendered HTML for both SendUpdate and WorkOrder):

1. Rendered HTML contains a top-level `<table bgcolor="#FAF8F5"` wrapper -- proves candidate (a) landed.
2. Rendered HTML's `<head>` contains the literal string `@media (prefers-color-scheme: dark)` -- proves candidate (b) landed.
3. Rendered HTML's `<head>` `@media` block contains at least one `!important` declaration -- proves the media-query rules use the same specificity-tie-resolver as the existing lock.
4. Rendered HTML's `<body>` element carries an inline `background:#FAF8F5` (or `background-color:#FAF8F5`) style attribute -- proves candidate (d) landed.
5. Rendered HTML's `<body>` element carries `bgcolor="#FAF8F5"` if the component supports the attribute (CONDITIONAL on react-email passthrough; if not, the (a) wrapper test covers it).

The existing 19 lock-marker assertions in `EmailShell.test.ts` (the 46.1-04 baseline) remain valid -- the chosen combination ADDS markers, it does not remove or reshape any existing one.

**Rationale -- why this combination over alternatives:**

- **Why not fewer techniques?** (a) alone may fail if Outlook-Mac auto-darken paints `<body>` higher up than `<table bgcolor>`. (d) alone may fail on Mac builds that override inline styles for auto-darken. (b) alone is `unknown` confidence -- mixed community evidence. Pairing (a) + (d) provides two HTML-attribute-level paints at different DOM levels (body and outer table); adding (b) provides a third path via media query for any Mac build that does honor `prefers-color-scheme`. Three independent paths cover three independent failure modes.
- **Why not more techniques?** (c) is rejected because it shares the failing delivery vehicle. Adding more `<style>`-block selectors does not help if the block itself is the failure surface. The literature review surfaced one additional candidate -- using `<center>` wrappers as a Mac-specific HTML4 fallback for centered layout -- which is documented as "Out-of-scope considered alternatives" because it's an alignment fix, not a color fix; gap-7 is color, not layout.
- **Why not replace the existing 46.1-04 lock?** D-11 explicitly forbids regressing the 19 existing lock-marker assertions. The chosen combination is purely ADDITIVE -- the `[data-ogsc]`/`[data-ogsb]`/MSO rules stay byte-identical; only the `<table bgcolor>` wrapper, the media query, and the inline body style are new.

**Open risks (round-4 Liz visual re-test will confirm or deny):**

- (a) survives Mac at `medium` confidence (Email on Acid is the lead source; Litmus corroborates). Round-4 is the proof. If round-4 still shows half-inversion, the next-round candidate is investigating whether react-email's `<Body>` is emitting an outer wrapper that prevents the `<table bgcolor>` from being top-level (unlikely given react-email's architecture, but worth verifying with a `console.log(html)` during 46.1-08 implementation).
- (b) is `unknown` for New Outlook for Mac. If round-4 is the proof, the spike's prediction is "(b) is the third-layer catch but probably does nothing -- (a) and (d) are doing the actual work." This is acceptable because (b) is cheap to add (one media query, no DOM changes) and provides upside on web M365 dark even if Mac ignores it.
- (d) at `medium` standalone confidence -- some Mac builds do override inline styles. Layering with (a) provides the second-layer survive. If round-4 STILL shows half-inversion despite (a)+(b)+(d) all landing, the failure mode is most likely "Mac auto-darken is painting at a level above all of these" -- a separate spike round would investigate whether VML or `mso-element` directives can paint above auto-darken on Mac (unlikely, but a known pattern in the deepest Outlook hardening playbooks).
- The `<table bgcolor>` wrapper introduces a new DOM layer. Snapshot diffs for SendUpdate and WorkOrder will be larger than 46.1-04's (one new layer of `<table>`/`<tr>`/`<td>` opens). 46.1-08 must verify the snapshot diff is wrapper-only (no inner content moved or reshaped).

**Archaeology disposition:** all three chosen techniques (a, b, d) received `archaeology PASS` per Step 4.5 above. No `### Archaeology finding` subheading required. 46.1-08 may apply the chosen combination without compensating attributes or selectors against the gap-5 / gap-6 left-alignment invariants.

## Regression-safety appendix

Explicit confirmation against each regression-safety axis the plan requires:

- **Does the chosen combination require removing any 46.1-04 lock rule?** No. Candidate (a) wraps; (b) appends a new media query inside the existing `<style>` block; (d) adds an inline `style=` prop on `<Body>`. The existing `[data-ogsc]` / `[data-ogsb]` / MSO rules stay byte-identical. The `<meta name="color-scheme" content="only light">` and `<meta name="supported-color-schemes" content="light">` tags stay byte-identical.
- **Are the 19 existing `EmailShell.test.ts` lock-marker assertions still valid?** Yes. All 19 assertions (across SendUpdate render path, WorkOrder render path, foundation hex parity) check for content the chosen combination does not remove or reshape: `[data-ogsc]` selector present, `[data-ogsb]` selector present, MSO conditional present, `mso-color-scheme: light` present, `!important` present, `#FAF8F5` inside `[data-ogsb]` rule, all 7 pill bg / 5 pill text / 7 pill border / 2 CTA / 2 foundation hexes present in rendered HTML. None of these are touched by adding a wrapper, a media query, or an inline body style.
- **Does the chosen combination regress Outlook-for-Windows light-mode behavior?** No. Per the findings table, (a) `survives (high)` for Outlook-Windows light (additive paint, no change to default light render); (b) is `not-applicable` for light mode (media query does not fire); (d) `survives (high)` for Outlook-Windows light. The existing 46.1-04 lock continues to work; the additions are silent in light mode.
- **Does the chosen combination regress Outlook web M365 dark-mode behavior?** No. Web M365 already honors `<meta name="color-scheme" content="only light">` and renders light. (a) is redundant paint; (b) reinforces the light palette via media query (web honors `prefers-color-scheme`); (d) is redundant inline paint. No regression risk.
- **Does Apple Mail dark-mode behavior change?** No. Apple Mail already honors the meta tag and renders light. The chosen combination adds redundant defenses; the rendering stays identical.
- **Does the chosen combination regress 46.1-05 + 46.1-07 left-alignment invariants?** No. Per the archaeology check above, all three chosen techniques are paint-or-CSS-only and do not introduce alignment-affecting wrappers. Inner `<td align="left">` attributes win over any outer table's `align`/`bgcolor`, so the `<table bgcolor>` wrapper from (a) does not override gap-5's left-alignment work in Procurement.tsx (4 cells) or Milestones.tsx (1 cell). When 46.1-07 lands its 3 additional `align="left"` Column attributes + 6 `width=` attributes + 3 `verticalAlign: "top"` body-row Columns + outer `<Section style={{ paddingTop: 16 }}>` wrapper, the same archaeology argument applies -- those are inner cell attributes; the (a) wrapper sits outside them.
