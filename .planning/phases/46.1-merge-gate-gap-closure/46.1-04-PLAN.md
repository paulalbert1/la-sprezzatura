**Why the catch-all `[data-ogsc] * { color: inherit !important; }` rule:**
The pill rules use class selectors (`.pill-scheduled` etc.), but the existing
section components apply pill chrome via INLINE styles, not className. The
class-selector rules above land in the `<style>` block but never match an
element in the rendered output -- so by themselves they're insufficient.
The `[data-ogsc] *` catch-all forces every text-bearing element under an
[data-ogsc]-tagged root to inherit its color from its parent (which gets
its color from the inline `style={{ color: "..." }}` already on the element).
This is the actual mechanism that prevents auto-darken from rewriting
inline-styled text. The class-selector rules are belt-and-suspenders for
any future migration toward className-driven chrome (and for the cream/CTA
backgrounds, which DO have classNames via Tailwind).

**Trade-off acknowledged:** the catch-all `* { color: inherit !important }` is
broad and load-bearing. It does NOT target backgrounds (different selector hook --
`[data-ogsb]`), so the per-pill bg locks above remain load-bearing. It DOES
forcibly override ALL descendant text colors -- including any inline
`style={{ color: "..." }}` on a leaf span -- because `!important` outranks
inline styles in CSS specificity. This is acceptable today because every
text-bearing leaf is reachable from a locked ancestor color: charcoal (`#2C2926`)
on headings, mid (`#4A4540`) on body and meta lines, stone (`#8A8478`) on
eyebrows/subline, the per-status pill colors via the catch-all parent. A
future template that needs a leaf-level accent color in Outlook desktop dark
mode CANNOT achieve it via inline `style={{ color }}` on the leaf -- the
catch-all will defeat that. Such a template MUST lock the accent via a
`[data-ogsc]` selector (class or attribute) on an ancestor element so the
selector-targeted color wins. Inline style on the leaf is insufficient.

Do NOT change anything else. Do NOT touch Tailwind config. Do NOT modify the
existing `<meta>` tags. Do NOT touch any section component.

The lock is mechanically Outlook-desktop-specific. |
| T-46.1-04-04 | T (Style override defeats a future intentional accent color in Outlook desktop) | `[data-ogsc] *{color:inherit!important}` catch-all | mitigate | The catch-all forces `color: inherit !important` on every descendant under `[data-ogsc]` -- inline `style={{ color }}` on a leaf span will be overridden, not respected (CSS `!important` outranks inline styles). This is acceptable today because every text-bearing leaf is reachable from a locked ancestor color. A future template that needs an accent color in Outlook desktop dark mode must lock that color via a `[data-ogsc]` selector (class or attribute) on an ancestor -- inline style on the leaf is insufficient. The plan's `<action>` documents this constraint so the next template author has a clear path. PR review for any future template adding leaf accent colors is the additional safeguard. |
| T-46.1-04-05 | D (Render performance -- bloated `<Head>` increases payload) | EmailShell.tsx `<style>` block size | accept | The new `<style>` block is ~3KB raw, ~1.2KB after gzip.
