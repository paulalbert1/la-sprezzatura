## Phase UI Scope

Phase 29 is primarily backend/architectural. The only **new** UI surface is the admin login page (`/admin/login`). All other work is refactoring existing admin pages and components to pass tenant context through -- no new visual components beyond login.

**New UI surfaces:**
1. Admin login page (`/admin/login`) -- email + password form
2. AdminNav brand section -- dynamic tenant `businessName` instead of hardcoded text

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing, nav item gaps |
| md | 16px | Default element spacing, form field margins |
| lg | 24px | Section padding, form group spacing |
| xl | 32px | Layout gaps, content area padding |
| 2xl | 48px | Major section breaks, sidebar vertical padding |
| 3xl | 64px | Page-level spacing |

Exceptions (inherited from existing portal login pattern in `src/components/portal/LoginForm.tsx` — intentional carry-forward, not new values):
- 12px (`py-3`) — input field vertical padding
- 20px (`px-5`) — alert/banner horizontal padding
- 40px (`px-10`) — submit button horizontal padding

**Source:** Existing admin layout uses Tailwind spacing utilities (`px-6` = 24px, `py-12` = 48px, `py-4` = 16px). The login form follows the portal login pattern which uses `px-5 py-4` (20px/16px) for alerts, `px-4 py-3` (16px/12px) for inputs, `px-10 py-4` (40px/16px) for submit button. This phase continues those established patterns rather than introducing new spacing.

**Accent reserved for:**
- Submit/primary action button background (`bg-terracotta`)

- Error validation text on form fields (`text-terracotta`)
- Error banner border (`border-terracotta/30`)

**Note:** This project uses terracotta as both accent and destructive/error color. The distinction is communicated through opacity and context (error banners use `terracotta/30` border + `terracotta/5` background), not through a separate destructive color. ## Component Inventory (Phase 29 Scope)

### New: Admin Login Page (`/admin/login.astro`)

**Layout:** Full-viewport centered form (no sidebar, no AdminLayout). Matches portal login pattern: centered card on cream background.

**Structure:**
```
<body class="bg-cream">
  <main> centered vertically and horizontally
    <brand mark> "La Sprezzatura" (later: tenant businessName) -- xs uppercase tracking
    <h1> "Studio Login" -- Display typography (Cormorant Garamond, text-4xl, font-normal)
    <subtitle> "Sign in to manage your projects" -- Body typography, stone color
    <AdminLoginForm client:load />
  </main>
</body>
```

**Form fields:**
1. Email input -- label "Email Address", type="email", autocomplete="email"
2. Password input -- label "Password", type="password", autocomplete="current-password"
3. Submit button -- "Sign In" (full width)

**States:**
| State | Visual Treatment |
|-------|-----------------|
| Idle | Both fields empty, button enabled |
| Submitting | Button shows "Signing in..." text, button disabled with `opacity-60` |
| Error (invalid credentials) | Error banner above form: terracotta/30 border, terracotta/5 background, "Invalid email or password. Please try again." |
| Error (rate limited) | Error banner: "Too many login attempts. Please wait a few minutes and try again." |
| Error (server) | Error banner: "Something went wrong. Please try again." |
| Field validation (email) | Inline error below field: "Please enter a valid email address" in `text-xs text-terracotta` |
| Field validation (password) | Inline error below field: "Please enter your password" in `text-xs text-terracotta` |
| Success | Redirect to `/admin/dashboard` (no success state shown on login page) |

**Input styling (matching portal login):**
```
bg-cream-dark text-charcoal border border-stone-light
px-4 py-3 text-sm font-body rounded
focus:outline-none focus:ring-2 focus:ring-terracotta focus:border-terracotta
placeholder:text-stone-light
[error state]: border-terracotta
```

**Submit button styling (matching portal login):**
```
bg-terracotta text-white text-xs uppercase tracking-widest font-body
px-10 py-4 w-full rounded-lg
hover:bg-terracotta-light transition-colors
disabled:opacity-60 disabled:cursor-not-allowed
```

## Interaction Contract

### Admin Login Flow

1. User navigates to any `/admin/*` route without a valid admin session
2. Middleware redirects to `/admin/login`
3. User enters email and password
4. Client-side validation: email format check on blur, password non-empty on submit
5. On submit: button enters disabled/submitting state, POST to `/api/admin/login`
6. On success (200): redirect to `/admin/dashboard` via `window.location.href`
7. On 401: show error banner "Invalid email or password. Please try again."
8. On 429: show error banner "Too many login attempts. Please wait a few minutes and try again."
9. On 500: show error banner "Something went wrong. Please try again."

**Keyboard accessibility:**
- Tab order: email field, password field, submit button
- Enter key submits form from either field
- Error banner receives `role="alert"` for screen reader announcement
- All inputs have associated `<label>` elements (not just placeholder text)
- Focus ring: `2px solid terracotta` with `3px offset` (from global.css `*:focus-visible`)

### Admin Logout

Existing `/admin/logout` link in sidebar and header. No changes needed.

## Accessibility Requirements

| Requirement | Implementation |
|-------------|---------------|
| Focus visibility | Global `*:focus-visible` rule: `2px solid terracotta, 3px offset` |
| Error announcement | All error banners use `role="alert"` |
| Form labels | Visible `<label>` elements with `htmlFor` binding to inputs |
| Color contrast | Charcoal (`#2C2926`) on cream (`#FAF8F5`): ratio 12.5:1 (AAA). Terracotta (`#C4836A`) on cream: ratio 3.8:1 (AA for large text, used only on buttons with white text or as accent). White on terracotta: ratio 4.1:1 (AA). |
| Password field | `autocomplete="current-password"` for password manager support |
| No motion dependency | Login page has no animations (unlike public site GSAP) |
| Responsive | Login form max-width 448px (`max-w-md`), centered, works on mobile |

## Responsive Behavior

| Breakpoint | Login Page | Admin Layout |
|------------|-----------|--------------|
| Mobile (< 768px) | Form stays centered, padding reduces to `px-6`, heading drops to `text-4xl` | Not applicable -- admin login is standalone; existing admin layout responsive behavior unchanged |
| Desktop (>= 768px) | Heading scales to `text-5xl` via `md:text-5xl` | No changes to existing 240px sidebar layout |

**Note:** The admin app (behind login) is designed for desktop use. The login page should be usable on mobile since an admin might need to log in from their phone, but the post-login admin experience is desktop-primary.
